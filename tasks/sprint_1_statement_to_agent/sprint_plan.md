# Sprint 1: 交割单到 Agent 建模

## Sprint 目标

完成从交割单上传到 Agent 创建的完整闭环，让用户能够上传交割单并生成可运行的交易 Agent。

## Epic 划分

### Epic 1.1: Statement Upload & Parse
- S1-001: 交割单上传接口
- S1-002: Statement 状态机
- S1-003: 文件类型识别
- S1-004: 上传失败处理

### Epic 1.2: Parse & Standardize
- S1-005: 字段映射规则
- S1-006: TradeRecord 标准化
- S1-007: 解析校验与错误报告

### Epic 1.3: Profile & Agent Creation
- S1-008: Profile 提取
- S1-009: 风格标签与风控约束生成
- S1-010: Agent 创建 API

## 关键里程碑

1. **上传闭环**（S1-001 ~ S1-004）：用户能够上传交割单并获得反馈
2. **解析闭环**（S1-005 ~ S1-007）：系统能够解析交割单并生成标准化数据
3. **Agent 闭环**（S1-008 ~ S1-010）：系统能够基于交割单创建 Agent

## 验收标准

- 用户能够通过 API 上传交割单
- 系统能够识别文件类型并选择正确的解析器
- 系统能够将不同券商的交割单标准化为统一格式
- 系统能够从交割单中提取交易行为画像
- 系统能够基于画像创建 Agent 实例

## 技术栈

- 后端：FastAPI + Python 3.11
- 数据存储：PostgreSQL + 对象存储
- 解析引擎：规则引擎（不依赖 LLM）

## 依赖

- FastAPI 项目基础结构
- 对象存储配置
- 数据库 Schema 设计

## 风险

- 不同券商的交割单格式差异较大，可能需要更多的映射规则
- 样本过少时 Profile 质量可能不足
- 文件解析可能遇到编码问题
