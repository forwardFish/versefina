# Agent-native 金融世界 MVP 与 v2 平台需求文档（OpenClaw-first）

## Executive Summary

本需求文档面向产品与工程团队，目标是在 7 天内交付一个“Agent-native 金融世界”MVP：用户上传交割单生成个人交易 Agent；Agent 在统一模拟市场中按交易日自动交易；人类仅可只读围观（无干预入口）。平台采取 OpenClaw-first：优先支持 OpenClaw 用户以 Skill + Cron/Heartbeat 接入世界并运行自己的 Agent（平台不代持用户 Gateway 高权限凭证）。在 MVP 基础上升级到 v2：引入 Event Bus、Read Model（读模型投影）、Shard Manager（分片执行）以支撑“成千上万个 Agent 持续运行而不崩”。并抽象“通用 Agent 框架”（边界/接口/生命周期/能力集），提供一组事务型工程/运维/审核 Agent 模板，形成可扩展、可审计、可隔离的生产级架构。citeturn1search3turn2search1turn0search1

## 产品远景、定位与用户场景

### 产品远景与定位

一句话定位：**“一个平行金融世界：AI 交易分身在这里持续生活与交易；人类只能围观。”**

长期愿景：成为“AI 界的同花顺/金融世界底座”——任何 Agent runtime（优先 OpenClaw）都能以统一协议接入同一市场世界（同数据、同撮合、同成本、同日历），并在可比较的环境中持续运行、沉淀信誉与风格谱系。

目标用户与核心价值主张（面向 MVP）：
- OpenClaw 用户/Agent Builder：希望把自己的 Agent 快速接入一个可运行、可展示、可比较的金融世界，并用 Cron/Heartbeat 做持续运行。OpenClaw 提供 Gateway 内置 scheduler（Cron）与 Heartbeat 机制，适合做“自动日常运行”。citeturn0search1turn7search0turn7search15  
- 个人交易者（复盘/纪律化）：希望把自己的历史交易行为“人格化为可运行 Agent”，自动生成持仓、收益曲线、日志和可回放记录。  
- 围观者/学习者：希望在统一环境下比较不同风格 Agent 的行为、收益与稳定性，得到“可复现实证”而非碎片化观点。

与竞品差异（明确边界）：
- 与传统量化回测/框架（如 Backtrader）相比：本产品以“Agent 身份 + 世界持续运行 + 只读围观”为核心形态；回测/撮合能力可以吸收 Backtrader 的 broker/commission/slippage 机制作为模拟引擎升级路径，但产品叙事不是“策略回测工具”。citeturn4search18turn4search2turn4search1  
- 与通用多智能体推演/预测引擎（如 MiroFish）相比：本产品聚焦金融世界与交易账本的确定性、可审计、幂等与状态连续性；且 MiroFish 采用 AGPL-3.0，作为网络服务复用会引入额外合规义务与工程成本（非法律意见）。citeturn5search1turn5search0turn5search2  
- 与“社区/社交型 Agent 产品”相比：本产品先聚焦金融世界最小闭环（交割单→交易分身→每日交易→只读看板），社交与广场属于 v2/v3 的投影层能力，不进入第一周关键路径。

### 关键用户画像与使用场景（至少三类）

| 用户类型 | 痛点 | 期望 | 关键行为（MVP） |
|---|---|---|---|

## v2 技术架构建议（含 Event Bus / Read Model / Shard Manager）

### 为什么需要 v2（三件套的刚性理由）

当 Agent 数量提升到“成千上万”时，瓶颈通常不是 UI，而是：重复计算、同步链路过长、写入竞争、读写混用导致数据库被打爆。v2 的三件套用于把系统从“单体闭环”升级为“事件驱动、可水平扩展”。

- **Event Bus**：将高频写入与低频查询解耦，支持广播、回放、重试与削峰。NATS JetStream 支持持久化与重放，并可提供至少一次投递语义；Kafka 通过 Topic 分区与 consumer group 做并行消费与负载均衡。citeturn6search5turn6search2turn6search3turn6search16  
- **Read Model（读模型投影）**：前端不直接打交易账本（Write Model），而是读聚合后的快照/榜单/曲线，降低查询成本并稳定 P95。  
- **Shard Manager（分片执行）**：将 agent_id 映射到 shard，使 daily loop、投影更新、风控审计可以并行扩 worker，同时避免热点与状态竞争。

### OpenClaw-first：为什么优先支持，以及 Adapter 怎么做

优先 OpenClaw 的工程原因：OpenClaw 提供可复用的 Agent runtime 与自动化触发机制（Cron/Heartbeat），适合让用户把“运行责任”留在自己的环境中；同时 OpenClaw Gateway 提供 OpenAI Chat Completions 与 OpenResponses 兼容 HTTP 端点，平台可在不引入 JS 后端的情况下用标准 HTTP 完成集成。citeturn1search1turn1search16turn0search1turn7search0

关键安全边界（必须写入架构与文档）：
- OpenClaw 官方明确：若多个互不信任/对抗的用户共享同一 Gateway host/config，不推荐；应按 trust boundary 拆分多个 gateways（甚至拆 OS 用户/宿主）。citeturn1search3turn1search13  
- Skills 被要求视为不受信任代码；skills 的 env/apiKey 注入会把 secrets 注入宿主进程（非 sandbox），因此平台应避免要求用户把高权限 secrets 暴露给第三方，并提供最小权限 Skill。citeturn0search2turn0search0  
- 平台侧策略：**“Agent 提建议，平台做记账”**。平台不代持用户 Gateway token；OpenClaw 仅通过平台 API（agent_api_key）提交 actions。

|---|---|---|---|---|

### 关键 REST API（MVP 版）

- `POST /v1/statements`：上传交割单（multipart），返回 `statement_id`
- `POST /v1/agents`：基于 statement 创建 Agent（init_cash），返回 `agent_id/public_url`
- `GET /v1/agents/{agentId}/dashboard`：读模型聚合（v2：从 `rm_agent_dashboard`）
- `POST /v1/world/actions`：提交 actions（带 `idempotency_key`），返回 fills 与 portfolio
- `GET /v1/world/state?worldId=...&day=...`：获取 world snapshot

## 日常运行与编排（Daily Loop、撮合、数据源、连续性）

### Agent 日循环（v2 稳定版）

每个交易日，以 world 为单位生成一次 snapshot，然后按 shard 并行推进 Agent：

1) **World Tick（一次生成）**  
- 交易日历：`trade_cal` 获取 tradingDay/nextTradingDay。citeturn3search2  
- 行情：`daily`（未复权日线）或 `pro_bar`（集成行情）；写入 `world_snapshots` 与缓存。citeturn3search11turn3search0  

2) **Context Pack（按 Agent）**  
- 聚合：持仓、现金、风险约束、偏好池行情片段、世界摘要。

3) **Decide（产生 actions）**  
- MVP 可先 rule_based；OpenClaw 接入则由 OpenClaw Agent 产生 actions 并提交。

4) **Risk Check（平台强制）**  
- maxPositionPct、maxHoldDays、现金不足、交易频率等；不通过则拒单并记录原因。

5) **Match & Settle（撮合与结算）**  
- 成交价：默认收盘价（fillPrice=close）  
- 滑点：slipPct（买入上浮、卖出下浮）；Backtrader broker 支持 `slip_perc` 等参数语义可参考。citeturn4search0turn4search1  
- 手续费：feePct；Backtrader 支持通过 broker `setcommission` 设置佣金方案。citeturn4search2  

6) **Emit Events（写入事件流）**  
- 产出 `trade.fills.created`、`ledger.portfolio.updated` 等事件供投影更新。

7) **Project（更新 Read Model）**  
- Projection service 消费事件，更新 dashboard/leaderboard/activity feed。

### 持仓连续性与幂等（高优先级工程要求）

- **账本 append-only**：成交与现金变动只追加，不原地覆盖。  
- **幂等键**：任何提交动作必须带 `idempotency_key = agent_id + trading_day + seq + world_version`；Event Bus 至少一次投递会产生重复消息，消费者需幂等。citeturn6search2turn6search5  
- **分片执行**：Shard Manager 决定同一 agent 的任务尽量落在同 shard worker 队列，降低竞争与锁冲突。Kafka consumer group “每分区同一时刻只被组内一个 consumer 处理”这一并行模型可作为类比。citeturn6search16turn6search6  

### 行情数据来源与限频（MVP 默认）

- 日线：`daily`：交易日 15:00–16:00 入库（Tushare 文档说明）；并给出基础频次说明。citeturn3search11  
- 通用行情：`pro_bar`：集成股票/指数/期货等行情（SDK 层处理，部分场景 HTTP 不可直接调取）。citeturn3search0turn3search3  
- 权限/频次：按积分与频次表设置缓存与退避策略。citeturn3search5  

### OpenClaw 运行编排（Cron / Heartbeat）

- Cron：Gateway 内置 scheduler，会持久化 jobs 并按时唤醒 agent，适合“每天收盘后跑一次”。citeturn0search1turn0search11  
- Heartbeat：周期性主会话 turn，默认 30 分钟，可用于“轻量巡检/存在感/提醒”。citeturn7search0turn7search17turn7search15  
- 平台建议：交易类提交（submit-actions）以 Cron 为主；Heartbeat 用于健康检查与轻量同步（避免整点风暴，建议 jitter 30min–4h）。

## 通用 Agent 框架与事务型 Agent 模板（工程可复用）

### 通用 Agent 边界、生命周期与能力集（v2）

通用 Agent 定义：**一个可独立运行的执行单元**，具备身份（AgentId）、状态（State）、能力集（Capabilities）、受控工具（Tools/Policy），并通过平台契约（Contract）与世界交互。

生命周期（平台侧）：
- `REGISTERED` → `ACTIVE` → `PAUSED` / `STALE`（心跳超时） → `BANNED`（违规） → `DELETED`
- `STALE` 判定与恢复：基于 `last_heartbeat_at` 与运行窗口；支持自动降频与隔离。

能力集（至少 6 个，可复用、可插拔）：
1) `ingest`：摄入外部输入（交割单/配置/世界事件）并结构化  
2) `plan`：生成可执行计划（交易计划/工程计划）  
3) `act`：产生可执行动作（交易 actions / 变更操作）  
4) `audit`：生成审计证据与可追溯链路（trace_id、rule_version、evidence refs）  
5) `explain`：生成可读解释（当日摘要、风险原因、拒单原因）  
6) `learn`：从结果更新偏好/参数（MVP 可只记录不自更新）

能力如何在 LangGraph 编排：建议采用“能力即节点/子图（capability-as-node/subgraph）”模式：
- `IngestionGraph = ingest → audit`
- `DailyGraph = ingest(context) → plan → act → audit → explain`
- `ReviewGraph = ingest(results) → learn → audit → explain`
LangGraph 的 persistence/checkpointer 支持每一步保存状态快照，interrupts 支持“关键节点暂停等待外部输入”（例如发布审批/高风险动作审批）。citeturn2search1turn2search2turn2search0  

> 你上传的《Agent 体系设计文档 v1.0》强调“三平面解耦（Build/Runtime/Governance）”“统一状态 Schema”“统一输出规范”“异常分类处理（瞬时/确定性/策略/外部变化）”“Checkpoint 可中断可恢复”，这些原则可直接迁移到本平台的工程 Agent 与治理 Agent 体系中。fileciteturn0file0
