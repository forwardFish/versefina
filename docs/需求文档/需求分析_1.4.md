# 需求分析_1.4.md

## 1. 文档定位

本版本为当前阶段的**主需求文档**，用于统一此前多份方案的方向分歧，并将项目收敛为一个**可落地、可验证、可逐步增强**的产品路线。

本版本的核心决策如下：

1. **主线正式调整为：金融叙事 / 题材扩散预演**
2. **第一阶段不再以“交割单镜像 Agent”作为主入口**
3. **交割单上传能力下放为第二阶段增强模块**
4. **借鉴 MiroFish 的“准备 → 运行 → 记录 → 报告”流水线**
5. **借鉴 Aaru 的“市场参与者仿真层 / 分布校准”思想，但不直接照搬其通用调研平台路线**
6. **第一版不做大而全的金融世界，不做自动交易，不做训练大模型，不做高频**

---

## 2. 一句话定义

**输入一条金融事件，系统输出：谁会信、谁会不信、题材会怎么扩散、明天看什么信号、什么情况下失效。**

---

## 3. 项目名称

**金融叙事 / 题材扩散预演引擎 MVP**  
副标题：**基于 Agent 分歧、Belief Graph、市场参与者仿真的事件沙盘系统**

---

## 4. 为什么要做这个产品

传统金融 AI 产品大多停留在两类：

1. **资讯总结型**：告诉你发生了什么
2. **单模型判断型**：告诉你可能利好/利空什么

它们的问题是：

- 无法把“市场参与者的预期”作为一等变量
- 无法表达参与者之间的共识、分歧、拥挤和证伪脆弱性
- 无法表达“预期 → 行为 → 价格 → 再预期”的反身性闭环
- 无法提前给出一套可验证的剧本，只能事后总结

本项目的目标不是直接预测涨跌，而是围绕事件构建一个最小可行的预演系统：

```text
事件
→ 参与者分歧
→ Belief Graph
→ 三剧本
→ 小规模模拟
→ T+1 / T+3 后验验证
→ 复盘报告
```

---

## 5. 当前版本的核心决策

### 5.1 这次主线为什么要调整

此前文档中有两条较强主线：

1. 交割单驱动的行为镜像 Agent
2. 事件驱动的金融叙事 / 题材扩散预演

本版本正式决定：

- **v1.4 的第一主线 = 事件驱动的题材扩散预演**
- **交割单镜像 Agent 不取消，但降级为后续增强层**

原因：

- 当前更容易形成可展示、可验证、可复盘的闭环
- 更接近用户真实高频需求：盘后、盘前、题材发酵判断
- 更适合先做出类似 MiroFish 的“事件沙盘”系统
- 相比“交割单→画像→镜像 Agent”，开发路径更短、理解成本更低、验证速度更快

### 5.2 交割单的正确位置

交割单在本版本中的作用不是“训练大模型”，而是：

- 提取真实交易行为特征
- 形成市场参与者风格母体
- 构建参与者画像与风格嵌入
- 为第二阶段的市场参与者仿真层做校准
- 作为后续分布级验证的重要数据资产

因此，交割单在 v1.4 中属于：

**Phase 2 / Phase 3 增强能力，而不是 P0 主入口。**

## 6. 用户与场景

### 6.1 核心用户

- A 股短线 / 事件驱动交易者
- 高频看资讯的研究员
- 财经内容创作者
- 中小投研团队
- 题材跟踪型自媒体 / 情绪交易者

### 6.2 典型使用场景

- 盘后看到一条政策消息，想判断明天如何发酵
- 看到上游原料涨价，想推演产业链扩散路径
- 某公司或行业出现负面消息，想推演利空扩散范围
- 想在盘前形成“剧本”，而不是只看碎片资讯
- 想知道某条逻辑为何可能成立，何时容易失效

## 7. 非目标

MVP 阶段明确不做以下内容：

- 不做自动交易
- 不接券商实盘
- 不做分钟级高频撮合
- 不做全市场全资产覆盖
- 不做开放 Agent 市场
- 不做复杂多用户协作权限体系
- 不做“完整 MiroFish 式世界图谱”
- 不做“训练自己的大模型”
- 不直接承诺“选股神器”或“涨跌预测神器”

## 8. 产品边界

### 8.1 当前产品本质

当前产品不是：

- 自动下单系统
- 通用投资顾问
- 大而全的金融数字孪生平台

当前产品是：

**单事件驱动的金融题材扩散预演系统**

### 8.2 第一版输出不是“买哪个票”

第一版输出应聚焦于：

- 哪条逻辑会不会发酵
- 由谁推动
- 先动什么
- 后动什么
- 什么情况下失效
- 哪些信号值得明天重点观察

---

## 9. 第一版事件范围

MVP 第一版只做一类事件，建议优先：

### 首选：涨价链 / 供给冲击 / 产业链传导

例如：

- 上游原料涨价
- 供给受限
- 限产 / 事故 / 停产
- 出口限制 / 进口约束
- 产业链成本传导

### 暂不优先

- 政策催化
- 利空扩散
- 跨市场复杂宏观事件

原因：

- 结构更清晰
- 更容易做事件结构化
- 更容易建立因果链和映射表
- 更适合题材扩散与板块传导分析

## 11. 系统核心思想

### 11.1 从“新闻总结”升级为“叙事预演”

旧范式：

```text
新闻
→ 总结
→ 利好 / 利空判断
→ 板块 / 个股
```

新范式：

```text
新闻
→ 结构化事件
→ 市场参与者分歧
→ Belief Graph
→ 预演扩散路径
→ 输出剧本
→ 后验验证
```

### 11.2 从“单模型判断”升级为“参与者分歧”

第一版不依赖一个万能模型直接拍结论，而是：

- 先抽取事件结构
- 再让不同参与者视角产生观点
- 把这些观点聚合成共识与分歧
- 再基于分歧生成多个可能剧本

### 11.3 从“单次正确”升级为“持续校准”

产品不是只做一次结论，而是持续积累：

- 哪类事件中哪类参与者更可靠
- 哪类分歧结构更容易演化成主线
- 哪类拥挤结构更容易崩
- 哪类信号最适合作为剧本失效条件

## 12. MVP 功能模块

### 12.1 Event Ingestion（事件输入）

#### 输入
- 用户粘贴新闻 / 公告 / 快讯
- 可选：来源链接、时间、备注

#### 输出
- 原始事件记录
- 事件 ID

#### 最小字段
- `event_id`
- `title`
- `body`
- `source`
- `event_time`
- `raw_text`

---

### 12.2 Event Structuring（事件结构化）

这是地基模块。

#### 目标
将原始新闻文本转为结构化事件 JSON。

#### 第一版必出字段
- `event_type`
- `entities`
- `sectors`
- `commodities`
- `affected_symbols`
- `causal_chain`
- `monitor_signals`
- `invalidation_conditions`

#### 说明
这一层必须产出稳定格式，而不是自由文本。

---

### 12.3 8 类 Agent / 参与者初始立场输出

第一版保留“8 类参与者视角”，但本质上更接近“功能 Agent + 简化参与者视角”。

#### 推荐 8 类视角
1. 游资 / 情绪资金
2. 公募 / 机构确认型
3. 产业研究 / 基本面视角
4. 政策研究 / 监管解读
5. 量化 / 风险预算视角
6. 风控 / 证伪视角
7. 媒体传播 / 情绪扩散视角
8. 产业链传导 / 供需链条视角

#### 每类 Agent 的统一输出协议
- `agent_type`
- `stance`
- `confidence`
- `time_horizon`
- `expected_impact`
- `evidence`
- `trigger_conditions`
- `invalidation_conditions`
- `first_movers`
- `secondary_movers`
- `dissent_points`

---

### 12.4 Belief Graph（共识 / 分歧图）

#### 目标
将 8 类参与者输出聚合为中间状态，而不是只保留一堆分析文本。

#### 第一版至少输出
- `consensus_strength`
- `dissent_width`
- `confidence_dispersion`
- `crowding_score`
- `falsification_fragility`
- `key_supporters`
- `key_opponents`

#### 注意
第一版可先用 Postgres + JSONB，不强制上图数据库。

---

### 12.5 Scenario Engine（三剧本引擎）

#### 目标
把共识 / 分歧转成可执行的明日剧本。

#### 必须输出
- `base_case`
- `bull_case`
- `bear_case`
- `dominant_scenario`
- `key_watchpoints`
- `invalidation_conditions`

#### 每个剧本至少包含
- 逻辑主线
- 先动对象
- 跟随对象
- 后续观察点
- 失效条件

---

### 12.6 Report Card（事件推演卡）

#### 第一版页面只需要一张卡
内容包括：
- 事件摘要
- 事件类型
- 8 类立场摘要
- 当前共识 / 分歧
- 三剧本
- 关键观察点
- 失效条件
- T+1 / T+3 验证状态

---

### 12.7 Outcome Tracker（后验验证）

这是闭环模块。

#### T+1 / T+3 需要回填
- 板块表现
- 龙头表现
- 产业链扩散情况
- 情绪扩散情况
- 哪个剧本最接近现实
- 哪类 Agent 的判断更有价值

#### 必须有的判分标签
- 命中
- 半命中
- 失效
- 延迟兑现
- 错方向

---

## 13. v1.4 借鉴 MiroFish 的部分

### 13.1 借鉴的不是“完整世界”，而是流水线

MiroFish 真正值得借鉴的是：

1. **先 prepare，再 run**
2. **先有角色，再有动作**
3. **动作要写日志**
4. **最后再做报告**

### 13.2 当前版本可直接借鉴的模块思想

#### 可借鉴 1：准备阶段
你的系统里对应为：

```text
事件输入
→ 事件结构化
→ 参与者立场生成
→ Belief Graph 初始状态
```

#### 可借鉴 2：运行阶段
你的系统里对应为：

```text
Scenario Engine
→ 小规模多轮模拟
→ 状态更新
```

#### 可借鉴 3：记录阶段
你的系统里对应为：

```text
actions.jsonl / simulation logs / 状态快照
```

#### 可借鉴 4：报告阶段
你的系统里对应为：

```text
Report Card / 复盘报告 / 追问接口
```

### 13.3 当前阶段不借鉴的部分

- 不做复杂世界图谱
- 不做社交平台双环境
- 不做长期记忆图谱回写
- 不做大规模多智能体社会
- 不做深度互动采访

## 14. 第二阶段：Market Participant Simulation Layer

这是 v1.4 的增强重点，不是 P0，但必须提前设计接口。

### 14.1 为什么要加这一层

否则会有两个断层：

#### 断层 A
交割单画像停留在“这个人过去怎么交易”，不能自然过渡到“新事件下这类人怎么反应”。

#### 断层 B
8 类 Agent 更像功能分析器，缺少真正的参与者异质性。

### 14.2 新增层的作用

位于：

```text
交割单 / 历史行为
→ 行为画像
→ Market Participant Archetypes
→ 事件推演系统
```

### 14.3 这一层的输出对象

#### MarketParticipantProfile
- `participant_id`
- `archetype_name`
- `participant_family`
- `style_embedding`
- `risk_budget_profile`
- `holding_preference`
- `reaction_rules`
- `invalidation_sensitivity`
- `authority_weight`

### 14.4 示例原型

#### Archetype: Fast Momentum 游资型
- 风格：追涨、偏高 Beta、重视板块合力
- 增强条件：龙头封板、成交放大、题材一致性增强
- 撤退条件：高开低走、龙头炸板、分歧急剧扩大

#### Archetype: Slow Confirmation 公募型
- 风格：中线确认、偏基本面、偏行业龙头
- 增强条件：政策覆盖范围明晰、产业逻辑兑现、估值可接受
- 撤退条件：落地慢于预期、业绩弹性不足、机构不跟

## 15. 交割单模块在 v1.4 中的定位

### 15.1 保留其价值，但调整顺序

交割单系统保留，但当前定位调整为：

- 不是主首页入口
- 不是第一阶段核心卖点
- 不是“训练大模型”的主语
- 是后续行为画像和参与者校准的数据来源

### 15.2 交割单在下一阶段的用途

1. 提取持仓周期、加减仓模式、追涨偏好、回撤容忍等行为特征
2. 形成 `MarketStyleEmbedding`
3. 生成参与者风格母体
4. 校准哪类事件更容易激活哪类参与者
5. 支持分布级验证

## 16. 题材映射与标的池字典（必须补）

这是当前最缺的基础资产之一。

事件进来以后，系统必须回答：

- 先影响哪个商品
- 再影响哪个行业
- 再影响哪些股票
- 龙头、跟风、补涨谁可能先动

因此必须维护一张基础映射表：

```text
商品
→ 产业链环节
→ 行业
→ A 股标的池
→ 风格标签（龙头 / 高弹性 / 机构票 / 情绪票）
```

### 16.1 第一版建议单独维护
文档名建议：

**《题材映射与标的池字典 v0.1.md》**

## 18. 阶段路线图与验收标准

### Phase 1：事件推演 MVP

#### 范围
- Event Ingestion
- Event Structuring
- 8 类 Agent 输出
- Belief Graph
- 三剧本
- T+1 / T+3 验证

#### 验收
- 至少支持 1 类事件稳定运行
- 至少支持 50 条真实样本
- 每条样本都能出完整事件卡
- 至少 30 条样本具备 T+1 / T+3 后验验证
- 至少 2 个指标优于“资讯总结 + 人工判断”基线

---

### Phase 2：参与者风格层

#### 范围
- Participant Profiles
- Participant Families
- 风格变体
- Belief Engine v2

#### 验收
- 同一事件可看到不同风格资金分歧
- Belief Graph 能表达“谁构成了共识”
- 能初步统计哪类参与者在何类事件中更可靠

---

### Phase 3：小规模模拟循环

#### 范围
- simulation_prepare
- simulation_runner
- action_logger
- state_updater

#### 验收
- 单个事件可跑 3~5 轮模拟
- 每轮均产生动作日志
- 状态会随轮次变化
- 剧本可动态修正
- 能输出“谁先动、谁跟随、谁撤退”的时序链

---

### Phase 4：复盘报告与追问

#### 范围
- retrieval_tools
- report_agent
- report_api

#### 验收
- 每个事件都有完整复盘报告
- 用户可追问“为什么”
- 报告能引用过程数据，而非空泛总结

---

## 22. 最终版本结论

v1.4 正式确定：

### 主线
**金融叙事 / 题材扩散预演**

### 方法
**事件驱动 + 参与者分歧 + Belief Graph + 三剧本 + 后验验证**

### 借鉴来源
- 借鉴 MiroFish 的流程型结构：准备、运行、记录、报告
