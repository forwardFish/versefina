# Versefina Graph-first 沙盘工作台重构说明.md

## 0. 文档定位

本文档不是讨论“Versefina 还缺什么功能”的泛化文档，而是一份**面向 Codex 的产品重构说明书**。

这份文档只解决一件事：

**把 Versefina 从“事件页增强版”重构为“Graph-first 的交易沙盘工作台”，让它在产品表达、页面中心、互动方式和演示效果上，尽可能逼近你参考的 MiroFish。**

这份文档将同时约束：

- 首页与工作台结构
- Graph Canvas 主舞台
- Trade Pulse Layer 交易脉冲层
- Decision Trace 决策链面板
- Report / Chat / Why 三个工作台
- 页面视觉与动线
- 旧页面的退居策略
- 后端接口和数据结构需要如何配合
- Codex 的实施顺序与验收方式

## 1. 为什么必须重构

根据当前 `roadmap_1_8` 完成总结，Versefina 已经完成：

- 事件输入与结构化
- population / clone / capital / influence 的对象基础
- `ExecutionActionV2` 与 trade ledger
- Influence Graph 进入决策驱动层
- Market State Engine V2
- `event-sandbox/[eventId]` 的四区布局基线
- replay / validation / scoreboards / why / counterfactual / calibration 等主链能力
- formal flow 与 authoritative closeout 收口到 completed boundary

也就是说，**Versefina 当前已经有“事件 → clone → 交易动作 → 影响网络 → 状态迁移”这条核心链。** fileciteturn38file3

但同一份完成总结也明确指出：
- 图谱页视觉张力还需要继续强化
- clone / scoreboard / calibration 仍有明显的产品体验打磨空间
- 当前属于“增强版单事件演化墙”，还不是“最终商业终局” fileciteturn38file3

这说明现在的问题不是“逻辑没做”，而是：

> **逻辑已经进入系统，但还没有被用“对的产品语言”表达出来。**

## 2. 参考对象 MiroFish 的产品语法，到底是什么

MiroFish 自己把系统定义成“基于多智能体技术的新一代 AI 预测引擎”，并把工作流写成五步：

1. 图谱构建  
2. 环境搭建  
3. 开始模拟  
4. 报告生成  
5. 深度互动 citeturn349577view0

这个定义非常重要。因为它说明，MiroFish 不是“先出报告，图谱只是附属物”，而是：

> **先造一个数字世界，再在世界里推演未来，最后才让 ReportAgent 去读这个世界并继续互动。**

而从你给的运行截图和仓库描述看，MiroFish 最有力量的地方，不只是功能，而是**产品重心的摆法**：

- 图谱可视化是主视角
- 节点详情 / 关系详情是浮层
- 右侧是阶段性环境准备 / agent 人设 /模拟输出
- 后续再切到报告工作台和深度互动工作台
- 用户始终感觉自己在观察一个“世界”，不是在看一个“表单 + 报表系统” citeturn349577view0

这就是 Versefina 现在要借鉴的核心，不是“长得像”，而是：

**产品中心从“结果页面”切换为“世界舞台”。**

## 3. 重构的唯一目标

本次重构的唯一目标是：

> **把 Versefina 的产品中心，从“事件页增强版”改成“Graph-first 的交易沙盘工作台”。**

具体说，就是：

- 图谱必须成为第一视角
- 交易必须成为页面上的一等公民
- 决策过程必须被产品化成“决策链”
- 报告 / 聊天 / Why 不再是几个孤立页面，而是“工作台态”
- 用户进入页面后，必须有“一个市场在动”的感受，而不是“一个系统在分析”

## 4. 重构后 Versefina 的最终产品形态

重构后，Versefina 不再以“事件详情页”为中心，
而是以一个新的页面壳层为中心：

### 4.1 新的产品中心页名称

建议统一命名为：

`/workbench/:eventId`

中文产品名称建议：
**沙盘工作台**

这个页面是整个系统的主入口。

旧的：
- `/event-sandbox/[eventId]`
- `/replay`
- `/participants/[participantId]`
- `/validation`

这些不删除，但不再是产品中心。  
它们退居为：

- 二级 drill-down 页
- 数据明细页
- 报告与验证页
- 历史回放与审计页

## 5. 新的页面总体结构

重构后的工作台必须由 **一个壳层 + 三种工作台态 + 两种视图模式** 组成。

### 5.1 顶层壳层（Workbench Shell）

整个页面必须先有一个稳定外壳，负责：

- 顶部导航
- 当前事件名称
- 当前工作台态
- 当前 Market State
- 当前 Round / 当前时间窗口
- 当前运行状态（Ready / Running / Completed）

#### 顶部左侧
显示：
- Versefina Logo
- 当前事件标题
- 事件类型标签
- 来源标签（finahunt / 手工输入 / 其他）

#### 顶部中部
显示三个切换按钮：
- 图谱
- 双栏
- 工作台

这部分是直接借鉴 MiroFish 截图里的“图谱 / 双栏 / 工作台”切换语法，但要服务 Versefina 自己的业务。你给的截图里 MiroFish 顶部就是这样切的。fileciteturn40file1

#### 顶部右侧
显示：
- 当前 Step / 当前模式
- 当前 Market State
- 当前 simulation 状态（Ready / Running / Completed）
- 最近更新时间

### 5.2 三种工作台态

#### 态 A：Graph / Sandbox
默认态，也是最重要的态。

目的：
- 看世界怎么动
- 看谁影响谁
- 看谁买卖了
- 看市场状态怎么变化

#### 态 B：Replay / Report
目的：
- 看结果如何被整理成报告
- 看成功链 / 失败链 / 脆弱链
- 看结构化验证结果

#### 态 C：Why / Interactive
目的：
- 对 clone、edge、scenario、state transition 深度追问
- 做反事实问题
- 看 evidence trace

### 5.3 两种视图模式

#### 模式 1：Graph-first
图谱占整页主舞台，右侧是 detail / panel。  
这是默认模式。

#### 模式 2：Dual-Pane
左侧图谱，右侧是准备过程 / 报告 / 聊天 / validation。  
用于更像 MiroFish 的双栏观察。

## 6. Graph-first 视图：必须怎么长

这是整个重构最核心的一层。

### 6.1 默认进入时，Graph Canvas 就是主舞台

进入工作台后，用户不应该先看到一堆卡片和表格。  
默认必须先看到一整张 Graph Canvas。

这张 Canvas 不是简单的关系图，而是一个“世界图”。

#### 这张图里至少同时存在 4 类节点

##### 节点组 1：事件节点
例如：
- 事件本身
- 关键商品
- 关键链条
- 风险因子
- 确认信号
- 失效信号

##### 节点组 2：市场对象节点
例如：
- 产业链环节
- 行业 / 板块
- 龙头标的
- 跟风标的
- 补涨标的

##### 节点组 3：参与者 family 节点
例如：
- 散户快钱
- 游资
- 公募 / 机构确认型
- 量化
- 产业链确认型
- 媒体扩散
- 风控
- 政策解读

##### 节点组 4：参与者 clone 节点
例如：
- `RetailFast_TrendChaser_02`
- `HotMoney_FastBreakout_01`
- `Institution_SlowConfirmation_01`

#### 这张图里至少同时存在 3 类边

##### 边 1：事件传导边
例如：
- commodity shock → sector
- sector → symbol
- confirmation signal → event strengthening
- invalidation signal → event weakening

##### 边 2：参与者影响边
例如：
- Media family → RetailFast family
- HotMoney clone → RetailFast clone
- ChainExpert clone → Institution clone
- Risk clone → Quant clone

##### 边 3：交易行为边
例如：
- clone → symbol 的 buy/add/reduce/exit
- clone → clone 的 influence-to-trade

#### 交付要求

Graph Canvas 必须支持：
- 拖拽
- 缩放
- 筛选层级
- 节点分类高亮
- 边类型高亮
- 当前时间窗口高亮
- 当前动作高亮

### 6.2 Graph Canvas 不能只是“全量乱图”

MiroFish 的截图虽然也有大图，但它不是一坨纯图。它有：
- 节点类别颜色
- 中心源头明显
- 右侧 detail panel
- 当前被点开的 node / relationship 详情浮层 fileciteturn40file1turn40file3

所以 Versefina 不能把图做成一张“技术炫图”。

必须分 3 层显示密度：

#### Level 1：默认聚合图
默认只显示：
- 事件节点
- 商品 / 板块 / 龙头
- family 节点
- family 级影响边

#### Level 2：family 展开
点开 family 后，展开 clone

#### Level 3：交易细粒度
在某个时间窗口，显示：
- clone 的交易动作边
- 当前买卖高亮
- 当前受影响 clone 高亮

也就是说：
**先有清晰结构，再允许深入，不是一上来就全量打满。**

## 7. Trade Pulse Layer：必须把“买卖”画出来

这是 Versefina 现在最缺、你最在意的一层。

你前面已经反复强调：
- 我不是只要他们讲话
- 我要看到他们在什么时间点买
- 买多少
- 如何因为其他 agent 的影响而买卖

所以必须正式新增：

### 7.1 Trade Pulse Layer（交易脉冲层）

这是一层叠在 Graph Canvas 上的动态层。

#### 它要表现什么

##### 1. 当前谁买了
某个 clone 执行 `INIT_BUY / ADD_BUY` 时：
- 节点发光
- 对应 symbol 边增亮
- 右侧出现一张 trade card

##### 2. 当前谁减了 / 退了
某个 clone 执行 `REDUCE / EXIT` 时：
- 节点闪退或变灰
- 原有边变细
- 风险提示边增强

##### 3. 当前这笔交易影响了谁
比如：
- `HotMoney_FastBreakout_01` 买入龙头 A
- 那么被它影响的 `RetailFast_TrendChaser_*` 周围会出现“待触发”高亮
- 右侧 detail panel 更新：
  - entry probability 上升
  - follow probability 上升

#### 视觉要求

Trade Pulse 不能藏在表里。  
必须是页面上的强视觉语言。

#### 最少要有的可视表现
- 节点发光 / 脉冲
- 交易边变粗
- 当前时间窗口色条
- symbol 热度增强
- 影响边联动高亮

### 7.2 交易动作卡（Trade Card）

每一笔关键交易，都必须有产品级 trade card，而不是只有日志行。

#### Trade Card 内容
- clone 名称
- family
- 时间窗口
- 动作类型
- 目标 symbol / theme
- 金额区间
- 仓位前后变化
- 触发原因
- 被谁影响
- 对市场状态的预期影响

#### 位置
默认出现在：
- 中央画布右上浮层
- 或右侧 detail panel 的顶部事件流

## 8. Decision Trace：要把“Agent 是怎么想的”做成决策链，不是长段解释

你说得非常对：你不是要长篇大论，而是要看 Agent 怎么一步一步被推着做出交易。

所以必须给每个 clone 增加一个产品化结构：

### 8.1 Decision Trace（决策链）

#### 点开 clone 后，必须看到以下层次

##### A. 当前身份
- family
- style_variant
- capital bucket
- 当前仓位范围
- 当前现金范围

##### B. 当前状态
- watching / constructive / skeptical / exiting
- 当前 confidence
- 当前 attention level

##### C. 当前看到的关键信号
例如：
- 龙头强
- breadth 扩大
- 价格持续
- 风控报警增强

##### D. 当前被哪些边影响
列出 top 3：
- 来自哪个 clone / family
- 哪条边
- 边强度
- 边在本轮如何影响了它

##### E. 当前动作决策链
例如：

`WATCH -> INIT_BUY`
- 原因 1：龙头强
- 原因 2：Media family 放大
- 原因 3：RetailFast 群体跟随预期增强

##### F. 对后续的预期
- 预计推动 market state 到哪里
- 预计会影响哪些 clone

#### 这层为什么重要

这层会把“AI 在思考”变成一种可见过程。

不是一大段解释，
而是：

**信号 → 受影响 → 决策切换 → 交易动作 → 状态影响**

## 9. 右侧 Detail Panel：必须像 MiroFish 那样成为“工作台信息区”

- 环境准备
- agent 人设卡
- 模拟参数
- 初始激活序列
- report 生成过程
- interactive tools 和 chat fileciteturn40file1turn40file3turn40file7

Versefina 必须学这一点，但要结合金融沙盘自己的业务。

### 9.1 右侧 Detail Panel 的 4 个主要标签页

#### 标签 1：Event / Seed
显示：
- 事件描述
- 结构化结果
- commodity / sector / symbol
- causal chain
- confirmation / invalidation
- lineage

#### 标签 2：Decision / Trade
显示：
- 当前选中 clone 的 decision trace
- 当前 trade cards
- 受影响对象
- 当前 market pulse

#### 标签 3：State / Scenario
显示：
- consensus_strength
- dissent_width
- crowding_score
- falsification_fragility
- dominant_scenario
- bull / base / bear
- 当前 state transition 解释

#### 标签 4：Why / Drilldown
显示：
- 点击 clone / edge / state / scenario 后的解释
- evidence trace
- replay 引用
- validation 引用

你现在已经有：

- why engine
- counterfactual

## 13. 后端与 API 层必须配合哪些改造

这次重构虽然是产品表达重构，但后端必须补这几类接口能力，才能支持 Graph-first 工作台。

### 13.1 Graph Stage 需要的接口

#### 统一图谱接口
`GET /api/v1/events/{event_id}/graph-stage`

返回：
- event nodes
- market object nodes
- family nodes
- clone nodes
- event edges
- influence edges
- trade edges
- current highlights

这样前端不用自己拼 5 个接口。

### 13.2 Trade Pulse 需要的接口

#### 当前时间窗口交易脉冲
`GET /api/v1/events/{event_id}/trade-pulse?round={round}&window={window}`

返回：
- 当前窗口关键动作
- 高亮 clone
- 高亮 symbol
- 关键 trade cards
- market pulse summary

### 13.3 Decision Trace 需要的接口

#### clone 决策链
`GET /api/v1/events/{event_id}/clones/{clone_id}/decision-trace?round={round}`

返回：
- clone profile
- current state
- seen signals
- influenced_by
- decision chain
- action executed
- expected impact

### 13.4 State Transition 解释接口

`GET /api/v1/events/{event_id}/market-state/transitions/{transition_id}`

返回：
- from_state
- to_state
- triggering clones
- triggering edges
- triggering signals
- evidence trace

### 13.5 Why / Interactive 接口
你现在已有 why / counterfactual 方向，但要收成：
- ask report
- ask graph
- ask clone
- ask state
- ask counterfactual

并能统一进右侧工作台。

## 14. 视觉风格要求（必须写死给 Codex）

你给的 MiroFish 截图整体有几个明显特点：

- 极简浅背景
- 黑白灰为主，少量高亮色
- 大量留白
- 中央图谱很大
- 右侧 panel 很规整
- 顶部控制很轻
- 底部 console / timeline 有“系统在运行”的感觉

Versefina 不需要照搬皮肤，但必须遵守这些“语言”：

### 14.1 必须保留的视觉原则
- 背景浅色
- 图谱区域最大化
- 边和节点高亮层次分明
- 面板不要太厚重
- 状态标签明确
- 时间窗口清晰
- 不要用杂乱彩色大卡片替代结构

### 14.2 必须避免的视觉错误
- 过多统计卡抢走主舞台
- 一屏塞满很多模块，没有层次
- 图谱缩成小块，只做装饰
- trade 信息只放在表格里
- 右侧 panel 信息堆叠，没有标签页

## 15. Codex 实施顺序（产品重构顺序）

虽然这份文档不是 phase 文档，但为了让 Codex 真正落地，仍然要给出实施顺序。

- WorkbenchShell
- Header
- Mode Tabs
- TimelineControl
- 一级路由 `/workbench/:eventId`
