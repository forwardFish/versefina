# 需求分析_1.7.md

## 0. 文档定位

本版本是在 `需求分析_1.6_最终版_事件参与者优先.md` 基础上的**动态增强版需求文档**。

1.6 解决的是：
- 主线切换
- 事件优先
- 参与者优先
- Belief Graph
- 三剧本
- T+1 / T+3 验证

但 1.6 仍然偏“静态盒子”：
- 能看见参与者各自观点
- 能看见聚合指标
- 能看见三剧本
- 但看不见参与者是如何一轮一轮被激活、如何互相影响、如何推动消息发酵、最后市场状态如何变化

**1.7 的任务，就是把系统从“静态分析器”升级成“动态演练器”。**

---

## 1. 一句话定义

**输入一条金融事件，系统不仅要输出各类参与者怎么看，还要输出：谁先被激活、谁影响谁、消息如何在多轮中发酵、哪些参与者推动了扩散、哪些参与者导致了失败、市场最终进入什么状态。**

## 2. 本版本核心升级

### 2.1 从“静态推演”升级为“动态参与者互动推演”

旧版本：

```text
事件输入
→ 事件结构化
→ 8类参与者输出
→ Belief Graph
→ 三剧本
→ T+1/T+3 验证
```

1.7 版本：

```text
事件输入
→ 事件结构化
→ Participant Roster 激活

→ Influence Graph 传播

→ Market State Machine
→ 三剧本动态修正
→ Simulation Output
→ T+1/T+3 验证
→ Report Card + Replay 页面
```

### 2.2 本版本真正要解决的问题

本版本不再只回答：

- 这条消息利好什么
- 哪些参与者支持这条逻辑
- 哪个剧本更可能成立

本版本还要回答：

- 这条消息最先被谁注意到
- 第一轮谁先行动
- 第二轮哪些参与者被带动
- 哪些参与者互相强化，形成共识
- 哪些参与者提出质疑，扩大分歧
- 哪个关键动作让市场从点火进入扩散
- 哪个关键动作让市场从扩散转为拥挤
- 哪个关键证伪条件导致崩塌
- 最终市场落在什么状态

## 8. Participant Roster 激活层

### 8.1 目标

Participant Roster 不再只是“有哪些参与者”，而是：

**一个消息进来后，哪些参与者会被率先激活、以什么权重进入模拟、初始状态如何。**

### 8.2 输入

- `event_type`
- `event_strength`
- `commodities`
- `chain_links`
- `sectors`
- `affected_symbols`
- `market_regime`
- `time_of_day`
- `source_quality`

### 8.3 输出

每个 participant 实例必须包含：

- `participant_id`
- `participant_family`
- `style_variant`
- `activation_score`
- `authority_weight`
- `risk_budget_profile`
- `initial_state`
- `initial_attention_level`
- `initial_confidence`
- `initial_action_candidates`

### 8.4 初始状态枚举

- `inactive`
- `aware`
- `watching`
- `constructive`
- `skeptical`

### 8.5 首发规则（P0）

#### 如果事件是 `supply_chain_price_shock`
优先激活：
- 产业链传导视角
- 游资 / 情绪资金
- 媒体传播 / 情绪扩散
- 机构确认型

次级激活：
- 量化 / 风险预算
- 风控 / 证伪

弱激活：
- 政策研究
- 宏观型解释者

### 8.6 激活的真实意义

这一层回答的问题是：

> “这条消息一出来，市场里谁先会看到，谁先觉得重要，谁先可能做出动作？”

## 9. Action Protocol（动作协议）

### 9.1 目标

定义参与者在每一轮里“到底能做什么”。

没有动作协议，就只有观点，没有模拟。

### 9.2 P0 动作集合

#### `IGNORE`
完全不理会该事件。

#### `WATCH`
观察，不下注、不传播。

#### `VALIDATE`
尝试找确认信号：
- 商品价格
- 龙头强度
- 板块联动
- 量能
- 产业链反馈

#### `BROADCAST_BULL`
向市场释放“看多 / 可发酵”的信息。

#### `BROADCAST_BEAR`
向市场释放“怀疑 / 风险高 / 逻辑不稳”的信息。

#### `LEAD`
成为首动推动者。

#### `FOLLOW`
在看到别人动作后跟随。

#### `EXIT`
从原有立场撤退，转为观望 / 风险提示。

### 9.3 每个动作需要的统一字段

- `action_id`
- `round`
- `participant_id`
- `action_type`
- `target_object`
- `confidence_delta`
- `broadcast_strength`
- `reason`
- `triggered_by`
- `expected_effect`

### 9.4 动作输出样例

```json
{
  "action_id": "act_001",
  "round": 2,
  "participant_id": "pf_fast_money_001",
  "action_type": "LEAD",
  "target_object": ["龙头A", "化工板块"],
  "confidence_delta": 0.18,
  "broadcast_strength": 0.82,
  "reason": ["龙头转强", "同链条二线跟随"],
  "triggered_by": ["media_bull_signal", "chain_confirmation"],
  "expected_effect": "推动市场从 ignition 进入 propagating"
}
```

### 9.5 动作协议的重要性

这一层回答的问题是：

> “参与者不是只会说，他在这一轮到底做了什么？”

## 10. Participant Interaction Engine（参与者互动引擎）

### 10.1 目标

把“多个参与者输出观点”升级为“多个参与者轮次互动”。

### 10.2 运行方式

每次模拟按 round 运行：

#### 每一轮做 5 件事
1. 读取上一轮 `MarketStateSnapshot`
2. 读取上一轮 `ActionLog`
3. 为每个活跃参与者生成本轮动作
4. 应用 `Influence_Graph`
5. 更新 `BeliefGraph` 与 `MarketState`

### 10.3 P0 建议轮次

- 默认 `3 ~ 5` 轮
- 第一版不做无限轮次
- 第一版不做开放式长时世界模拟

### 10.4 每轮更新对象

- `participant_states`
- `participant_confidences`
- `participant_actions`
- `influence_events`
- `belief_snapshot`
- `market_state_snapshot`

### 10.5 参与者互动最关键的设计原则

#### 原则 1：不是每个参与者都每轮行动
有些参与者只在信号确认后才行动。

#### 原则 2：不是每个参与者都同权
不同参与者对市场的传播与影响权重不同。

#### 原则 3：行动会改变别人的行动概率
这才是“互动”的核心。

#### 原则 4：最终市场状态是互动结果，不是单次判断结果

## 11. Influence Graph（影响传播图）

### 11.1 目标

定义：
**谁影响谁、影响多大、影响要等几轮、在什么条件下影响失效。**

### 11.2 数据结构

#### `participant_influence_edge`
- `source_participant`
- `target_participant`
- `influence_type`
- `weight`
- `lag_rounds`
- `activation_condition`
- `expiration_condition`

### 11.3 P0 典型边

#### 媒体传播 → 游资
- `influence_type = narrative_amplification`
- `weight = 0.7`
- `lag_rounds = 0`

#### 游资首动 → 跟风资金
- `influence_type = momentum_confirmation`
- `weight = 0.9`
- `lag_rounds = 0`

#### 产业链确认 → 机构确认型
- `influence_type = fundamental_validation`
- `weight = 0.8`
- `lag_rounds = 1`

#### 风控 / 证伪 → 量化
- `influence_type = risk_signal`
- `weight = 0.6`
- `lag_rounds = 0`

### 11.4 P0 影响规则示例

- 若媒体传播 `BROADCAST_BULL` 强度 > 0.7，则游资 `WATCH -> LEAD/FOLLOW` 概率提升
- 若龙头确认 + 产业链确认同时出现，则机构确认型 `WATCH -> VALIDATE/CONSTRUCTIVE`
- 若风控发出强证伪信号且龙头失去强度，则量化 `FOLLOW -> EXIT`

### 11.5 Influence Graph 的真实作用

这一层回答的问题是：

> “不是 8 个参与者各说各话，而是谁把谁带动了。”

## 12. Belief Graph 动态更新层

### 12.1 目标

Belief Graph 不再只是静态聚合，而要每轮更新。

### 12.2 每轮更新字段

- `consensus_strength`
- `dissent_width`
- `confidence_dispersion`
- `crowding_score`
- `falsification_fragility`
- `key_supporters`
- `key_opponents`
- `dominant_support_chain`
- `dominant_opposition_chain`

### 12.3 动态更新原则

#### 如果：
- `LEAD` 和 `FOLLOW` 增多
- key supporters 增多
- confirmatory signals 增强

则：
- `consensus_strength` 上升
- `dissent_width` 下降

#### 如果：
- 看多与怀疑动作同时增加
- 参与者间分歧扩大
- 机构与快钱方向相反

则：
- `dissent_width` 上升
- `confidence_dispersion` 上升

#### 如果：
- 支持者越来越同质
- 追涨类动作占比过高
- 风控参与度下降

则：
- `crowding_score` 上升
- `falsification_fragility` 上升

### 12.4 新增动态字段

建议新增：

- `belief_momentum`
- `support_chain_density`
- `opposition_chain_density`
- `narrative_temperature`

## 13. Market State Machine（市场状态机）

### 13.1 目标

明确一个消息在多轮互动后，市场最终处于什么状态。

### 13.2 P0 状态集合

#### `DORMANT`
消息存在，但没有形成市场主线。

#### `IGNITION`
已有首动参与者，主线开始点火。

#### `PROPAGATING`
跟风扩散出现，板块 / 链条开始联动。

#### `CROWDED`
一致性过高，参与者结构开始拥挤。

#### `FRAGILE`
逻辑看似强，但对少数关键验证条件依赖很高。

#### `INVALIDATED`
关键失效条件触发，主线被证伪。

### 13.3 状态切换规则（P0）

#### `DORMANT -> IGNITION`
条件：
- `first_movers` 出现
- `consensus_strength` 上升
- 至少一个强确认动作出现

#### `IGNITION -> PROPAGATING`
条件：
- `secondary_movers` 扩大
- 板块 breadth 扩大
- `dissent_width` 缩小

#### `PROPAGATING -> CROWDED`
条件：
- `crowding_score` 超阈值
- 参与者构成过度同质
- follow 动作占比过高

#### `PROPAGATING -> FRAGILE`
条件：
- 高一致但验证点单一
- `falsification_fragility` 上升

#### `CROWDED -> INVALIDATED`
条件：
- 关键失效信号触发
- 龙头转弱
- 风控链条扩散

#### `FRAGILE -> INVALIDATED`
条件：
- 关键证伪条件兑现
- 支撑链断裂

### 13.4 状态机存在的意义

这一层回答的问题是：

> “消息经过多轮参与者影响以后，市场到底变成什么样了？”

## 14. Scenario Reviser（三剧本动态修正器）

### 14.1 目标

三剧本不能只在第 0 轮生成一次，而要随着模拟过程更新。

### 14.2 修正时机

每轮结束后，根据：
- Belief Graph 动态变化
- MarketState
- 新动作日志
- 新确认 / 失效信号

修正：
- `base_case`
- `bull_case`
- `bear_case`
- `dominant_scenario`

### 14.3 P0 修正规则示例

#### 若 market_state 进入 `PROPAGATING`
- 提高 `bull_case` 权重
- 强化 first_movers / secondary_movers 链

#### 若 market_state 进入 `FRAGILE`
- base_case 下调
- bear_case 上调
- invalidation_conditions 强调

#### 若 market_state 进入 `INVALIDATED`
- dominant_scenario 切换为 bear_case
- 记录 failure chain

### 14.4 这一层的作用

这一层回答的问题是：

> “剧本不是死的，而是随着市场反应不断修正的。”

## 19. 前端交互要求

### 19.1 必须支持“轮次切换”
用户不能只看到最终结论，要能切换 Round 1 / 2 / 3 / 4。

### 19.2 必须支持“点击某个参与者看详情”
否则“参与者如何参与”无法真正被感知。

### 19.3 必须支持“点击影响线看原因”
用户要知道：
- 为什么 A 影响了 B
- 基于哪个动作
- 强度多大

### 19.4 必须支持“时间轴回放”
否则过程感不够。

### 19.5 必须支持“Belief Graph 与 Market State 联动”
当用户切换轮次时：
- Belief 指标同步变化
- MarketState 同步变化
- 剧本同步变化

## 25. 最终结论

### 25.1 1.7 版真正补上的东西

1. 不再只是“各个参与者怎么看”
2. 而是“各个参与者怎么一轮一轮行动”
3. 不再只是“Belief Graph 是什么”
4. 而是“Belief Graph 如何随着互动变化”
5. 不再只是“三剧本怎么生成”
6. 而是“三剧本如何在动态演练中被修正”
7. 不再只是“最终输出报告”
8. 而是“过程也必须能在页面中被看到”

### 25.2 一句话总结

**1.7 的目标，是把你的系统从“会分析消息”升级成“会把消息放进一个金融参与者沙盘里，演出它如何发酵、如何扩散、如何失败，并把全过程展示给你看”。**
