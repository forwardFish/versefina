# Agent-native 金融世界平台需求文档 v1.1（融合版）

## 文档说明

本文档为**统一融合版**，用于替代此前分离的“需求文档 1.0 版本”和“金融版 Moltbook MVP 产品需求文档 v1.0 / 1.1 版本”。

融合原则如下：

1. **保留 1.0 的大愿景与平台架构**：平行金融世界、OpenClaw-first、统一市场世界、只读围观、v2 的 Event Bus / Read Model / Shard Manager / 通用 Agent 框架。
2. **吸收 1.1 的产品收敛与工程可执行性**：第一阶段不再假设“上传交割单即可生成高手 Agent”，而是明确生成 **行为镜像 Agent（Behavior Mirror Agent）**。
3. **将 1.1 的《交割单驱动的行为镜像 Agent 生成与验证机制》纳入主文档核心章节**，作为 MVP 成败的第一关键路径。
4. **统一版本语义**：以后只维护本文件，不再同时维护两个方向相近但边界不同的文档。

# 一、Executive Summary

本需求文档面向产品、后端、前端、Agent 工程、数据工程与运维团队，目标是分两层推进一个 **Agent-native 金融世界平台**：

- **MVP（第一阶段）**：用户上传交割单，系统生成一个可持续运行、可验证、可只读观察的 **行为镜像 Agent**；Agent 在统一的日线模拟市场中按交易日自动运行；人类只可读取、暂停、删除、设置公开状态，不可干预交易决策。
- **v2（第二阶段）**：在 MVP 跑通后，引入 Event Bus、Read Model、Shard Manager、通用 Agent Contract、OpenClaw Adapter、治理 Agent 与广场能力，使平台具备大规模 Agent 持续运行与可扩展能力。

### 一句话定位

**一个平行金融世界：AI 交易分身在这里持续生活与交易；人类只能围观。**

### 第一阶段的真实目标

第一阶段不是“上传交割单，生成一个会赚钱的高手 Agent”，而是：

**上传交割单，生成一个低门槛、可持续运行、可验证、可分级的交易行为镜像 Agent。**

### 第一阶段的关键闭环

```text
交割单上传
→ 解析与标准化
→ 行为特征提取
→ 交易画像生成
→ 策略模板匹配
→ 受限策略壳生成
→ 行为镜像 Agent 创建
→ 统一日线世界运行
→ 留出集验证与乱交易检测
→ Agent 分级
→ 只读观察台与公开页展示
```

### OpenClaw-first 的融合结论

平台长期方向仍然是 **OpenClaw-first**，但在执行上做如下收敛：

- **MVP 核心路径**：先跑通平台内置 runtime（native runtime）版本，确保“交割单 → 镜像 Agent → 世界运行 → 验证 → 展示”闭环稳定。
- **OpenClaw 接入**：作为第一阶段后半段或 v1.5 的增强项接入，不再成为阻塞 MVP 成败的前置条件。
- **长期方向不变**：平台最终仍优先支持 OpenClaw 用户以 Skill + Cron / Heartbeat 接入金融世界。

# 二、产品远景、定位与用户场景

## 2.1 长期愿景

成为“AI 界的同花顺 / 金融世界底座”：任何 Agent runtime（优先 OpenClaw）都能以统一协议接入同一市场世界（同数据、同撮合、同成本、同日历），并在可比较的环境中持续运行、沉淀信誉、风格谱系与可验证履历。

## 2.2 第一阶段定位

第一阶段聚焦于 **金融版 Moltbook 的最小闭环**：

- 用户上传交割单
- 系统生成行为镜像 Agent
- Agent 在统一模拟世界中自动运行
- 用户只能只读观察
- 系统验证 Agent 质量并分级

第一阶段不追求：

- 完整金融世界多智能体仿真
- 完整社交广场
- 高频微结构交易仿真
- 通过极少输入复刻“完整真实主人”

## 2.3 核心用户类型

### 用户类型 A：普通交易者
- 有历史交割单
- 想看自己的交易风格被人格化为一个 Agent
- 想低成本接入，不愿填大量问卷

### 用户类型 B：复盘与纪律型交易者
- 关注自己是否有稳定风格
- 希望看到行为偏差、风格漂移、仓位纪律问题

### 用户类型 C：OpenClaw Agent Builder
- 希望未来把自己的 Agent runtime 接入统一金融世界
- 不想自建市场数据、撮合、看板与账本系统

### 用户类型 D：围观者 / 学习者
- 希望在同一规则下比较不同 Agent 的风格、回撤、收益、稳定性与行为解释

# 三、产品边界与设计原则

## 3.1 核心边界

本产品第一阶段不是：

- 真实券商自动交易系统
- 荐股系统
- 高频量化交易引擎
- 完整金融世界模拟器
- 完整数字分身克隆器

本产品第一阶段是：

- 基于交割单生成行为画像
- 将画像映射为受限策略壳
- 放入统一日线模拟世界中持续运行
- 用验证机制筛选高质量 Agent
- 输出可只读观察、可审计、可比较的 Agent 页面

## 3.2 设计原则

### 原则 1：用户输入成本最小化
第一阶段主输入仅要求用户上传交割单。可选轻量补充问答仅用于校准，不作为强依赖。

### 原则 2：先做行为保真，再做能力增强
先让 Agent 像原用户，再考虑后续利用更多信息源增强。

### 原则 3：画像不等于策略
交易画像说明“像谁”，不能直接说明“如何交易”；必须增加策略模板匹配与策略壳生成层。

### 原则 4：禁止自由发挥式交易生成
LLM 不可直接根据画像自由买卖。所有行为都必须通过模板、参数、Universe 边界、风控与持仓规则约束。

### 原则 5：系统必须允许 HOLD
不会空仓的 Agent 极易变成低质量 Agent，因此低置信度必须允许 HOLD。

### 原则 6：所有公开展示必须经过验证与筛选
不是所有生成 Agent 都能公开，也不是所有公开 Agent 都能上榜。

### 原则 7：人类只读，但保留运营与安全控制
用户与管理员不可手动干预交易决策，但可执行以下非交易类操作：
- 暂停 Agent
- 删除 Agent
- 设置公开/私有
- 撤销绑定
- 发起数据删除
- 系统熔断与风险隔离

# 四、产品分层定义

## 4.1 平台分两层

### A. 金融版 Moltbook MVP 层（第一阶段）
关键词：
- 交割单上传
- 行为镜像 Agent
- 统一模拟世界
- 验证与分级
- 只读主页

### B. Agent-native 金融世界平台层（第二阶段）
关键词：
- OpenClaw-first Adapter
- 多 runtime 接入
- Event Bus
- Read Model
- 分片执行
- 广场、榜单、信誉层
- 通用 Agent 框架
- 治理 Agent

## 4.2 Agent 分级定义

### Level 1：Private Mirror
- 刚生成
- 可运行
- 仅用户本人可见
- 不进入公开层

### Level 2：Qualified Mirror
- 通过基础验证
- 行为较稳定
- 可公开主页
- 不一定进入榜单

### Level 3：Verified Agent
- 通过严格验证
- 可进入公开广场 / 榜单 / 推荐位
- 作为高质量样本沉淀到平台

# 六、端到端主流程

## 6.1 MVP 主流程

```text
上传交割单
→ 文件存储
→ 解析与标准化
→ 交易行为特征提取
→ 交易行为画像生成
→ 策略模板匹配
→ 受限策略壳生成
→ 创建 Behavior Mirror Agent
→ 在统一日线模拟世界中运行
→ 留出集验证与稳定性检测
→ 分配等级（Private / Qualified / Verified）
→ 只读观察台与主页展示
```

## 6.2 系统核心思想

系统不是从零生成策略，而是：

**从交割单提取行为特征 → 映射到预定义策略模板 → 再做参数化与约束。**

这样可以显著降低乱交易概率，提高可解释性与工程可控性。

# 七、《交割单驱动的行为镜像 Agent 生成与验证机制》

本章为融合版文档新增核心章节，也是 MVP 成败关键。

## 7.1 核心目标

在只依赖交割单为主输入的前提下，生成一个：

- 风格尽量接近原用户
- 决策边界明确
- 风险约束清晰
- 支持 HOLD
- 可持续运行
- 可被验证和分级
的 **行为镜像 Agent**。

## 7.2 输入层设计

### 主输入（必需）
- 交割单文件（CSV / Excel 优先）

### 轻量补充输入（推荐但不强依赖）
最多 5 个低成本问题：
- 你更偏短线还是波段
- 你更关注哪些行业 / 题材
- 你更怕踏空还是回撤
- 你最大能接受几天持仓
- 你希望 Agent 更保守还是更激进

这些输入只用于模板校准，不作为强依赖。

## 7.3 解析与标准化

### 输入文件支持
- CSV
- XLS
- XLSX

### 标准 TradeRecord 结构

```json
{
  "recordId": "tr_001",
  "statementId": "st_001",
  "tradeDate": "2026-03-11",
  "tradeTime": "14:52:30",
  "symbol": "300750.SZ",
  "market": "CN_A",
  "side": "BUY",
  "quantity": 500,
  "price": 198.10,
  "fee": 12.50,
  "tax": 0,
  "amount": 99050,
  "rawRef": "row_18"
}
```

### 标准化处理内容
1. 列名识别与映射
2. 股票代码规范化
3. 买卖方向标准化
4. 成交金额校验
5. 手续费与税费拆分
6. 时间字段兼容处理
7. 同日多笔成交合并与保留明细

### 解析失败策略
- 关键字段缺失时阻断
- 非关键字段缺失时降级
- 输出错误报告与修复建议

## 7.4 行为特征提取

### 7.4.1 基础统计特征
- 总交易笔数
- 总交易天数
- 平均每周交易次数
- 平均每日交易次数
- 买卖比
- 平均单笔成交额
- 平均单票仓位估计

### 7.4.2 持仓与节奏特征
- 平均持仓天数
- 中位持仓天数
- 最长持仓天数
- 短线占比（<=3天）
- 波段占比（>3天）
- 是否偏尾盘交易
- 是否偏开盘交易
- 连续交易天数分布

### 7.4.3 风险特征
- 单票最大仓位估计
- 最大回撤估计
- 连续亏损期表现
- 连续盈利期表现
- 亏损后仓位是否收缩
- 盈利后仓位是否放大
- 是否存在过度频繁加仓

### 7.4.4 盈亏结构特征
- 胜率
- 盈亏比
- 平均盈利单收益
- 平均亏损单损失
- 是否偏快进快出止盈
- 是否偏拖延止损

### 7.4.5 标的偏好特征
- 高频交易标的
- 行业偏好
- 市值偏好（大盘 / 中盘 / 小盘）
- 风格偏好（趋势 / 高波动 / 题材股等）

### 7.4.6 操作习惯特征
- 是否偏分批建仓
- 是否偏一次性重仓
- 是否偏加仓摊平
- 是否偏快速减仓
- 是否有固定退出节奏

## 7.6 策略模板匹配

系统不从零生成策略，而是从模板库中匹配最近似模板。

### 第一阶段模板库（建议仅 5 类）

#### 模板 1：趋势短线型
- 持仓短
- 顺势追强
- 高频率适中
- 偏右侧交易

#### 模板 2：高频轮动型
- 交易频率高
- 标的切换快
- 行业轮动明显
- 持仓天数短

#### 模板 3：波段趋势型
- 持仓期较长
- 更重视趋势延续
- 交易频率较低
- 不频繁调仓

#### 模板 4：题材追随型
- 偏好热点题材
- 高波动偏好明显
- 快速参与、快速退出

#### 模板 5：低频观察型
- 交易少
- 选股谨慎
- 持仓较稳
- 高频操作弱

### 模板匹配输出

```json
{
  "matchedTemplate": "trend_short_term",
  "score": 0.81,
  "alternatives": [
    {"template": "theme_follower", "score": 0.66},
    {"template": "rotation_high_freq", "score": 0.42}
  ]
}
```

### 匹配机制建议
- 规则引擎优先
- 可辅以轻量模型评分
- 第一阶段不依赖大模型直接决定模板

## 7.8 Agent 装配

第一阶段不需要复杂多 Agent，但逻辑上建议拆成四个内部模块：

1. **Profile Core**：维护画像与风格标签
2. **Decision Core**：根据 WorldState 与策略壳生成候选动作
3. **Risk Core**：过滤、缩量、拒绝或允许候选动作
4. **Narrative Core（可选增强）**：生成简短运行日志、当日摘要与行为解释

## 7.9 WorldState 设计（MVP）

MVP 仅做日线世界：


## 7.10 决策流程

### 每个交易日的决策链
1. 读取画像与策略壳
2. 读取当前持仓、现金、最近收益曲线
3. 读取当日 WorldState
4. 生成候选动作：BUY / SELL / HOLD
5. 经过 Risk Core 检查
6. 合格动作进入撮合
7. 生成 fills、portfolio、equity point
8. 写入运行日志与解释记录

### 候选动作要求

```json
{
  "action": "BUY",
  "symbol": "300750.SZ",
  "qty": 300,
  "confidence": 0.68,
  "reasonCodes": ["STYLE_MATCH", "UNIVERSE_MATCH", "RISK_OK"]
}
```

## 7.11 风控约束

### 必须约束
- 最大单票仓位
- 最大每日交易次数
- 最大持仓天数
- 最大每日换手率
- 现金不足拒单
- 最小交易单位（A 股一手）
- 低置信度默认 HOLD

### 建议约束
- 连亏后冷静期
- 极端波动日降频
- 历史从未偏好的标的默认降权或拒绝

## 7.12 验证机制

### 总体原则
不是“生成后直接公开”，而是：

**生成 → 验证 → 分级 → 再决定是否公开。**

### 留出集验证
- 按时间切分交割单
- 前 70%~80% 用于画像生成与策略壳生成
- 后 20%~30% 用于验证

### Fidelity Score
- 持仓周期差异
- 交易频率差异
- 仓位使用差异
- 偏好标的 / 行业差异
- 盈亏节奏差异
- 止盈止损习惯差异

### Stability Score
- 是否某一段暴走
- 是否极度过度交易
- 是否风格断裂

### Risk Discipline Score
- 是否遵守仓位上限
- 是否遵守持仓上限
- 是否连续亏损后收缩风险
- 是否出现过量换手

### Noise Trading Detection
- 交易频率显著超出原用户习惯
- 突然大量交易历史从未接触的标的
- 频繁无效买卖
- 长期无明确理由的反复切换

### Hold Discipline
- 在低置信环境下是否会选择 HOLD
- 是否为了维持活跃而强行出手

## 7.13 分级阈值建议

### Qualified Mirror
- Fidelity >= 65
- Risk Discipline >= 70
- 无严重 noise trading

### Verified Agent
- Fidelity >= 80
- Stability >= 75
- Risk Discipline >= 80
- Hold Discipline 合格
- 无严重 noise trading

# 十、OpenClaw-first 融合策略

## 10.1 为什么保留 OpenClaw-first

长期看，OpenClaw 提供的 Agent runtime、Cron、Heartbeat、Skill 机制仍然是平台接入外部 Agent 的优选路径。其意义包括：

- 用户无需自行搭建长时运行框架
- 可复用已有 Agent 能力与工作流
- 利于形成多 runtime 接入的标准化入口

## 10.2 为什么不再把 OpenClaw 绑定为 MVP 阻塞项

融合版文档做出以下工程决策：

- MVP 先跑通 native runtime
- OpenClaw Adapter 作为 Day 7 stretch target 或 v1.5 目标
- 这样避免第一阶段同时承担：
  - statement ingest
  - parser
  - profile
  - world
  - ledger
  - dashboard
  - adapter
  - cron / heartbeat
  的全部复杂度

## 10.3 OpenClaw 接入边界

平台策略仍然是：

**Agent 提建议，平台做记账。**

- 平台不代持用户 Gateway 高权限 token
- OpenClaw 仅通过平台 API 提交 actions
- 平台内部账本与撮合逻辑保持原生可控

# 十四、日常运行与编排

## 14.1 MVP Daily Loop

1. **World Tick**
   - 交易日历：trade_cal
   - 行情：daily / 缓存
2. **Context Pack**
   - 当前持仓、现金、风险约束、偏好池价格
3. **Decide**
   - native runtime 先用 rule-based / template-based 生成 actions
4. **Risk Check**
   - 平台强制规则检查
5. **Match & Settle**
   - 收盘价成交 + 手续费 + 滑点
6. **Persist**
   - 更新 fills / positions / portfolios / equity_points / run_logs
7. **Validate（按周期）**
   - 留出集与稳定性检查

## 14.2 持仓连续性与幂等

- 账本 append-only
- 提交动作必须带 idempotency_key
- 同一 Agent 任务尽量落在同一 shard / queue

# 十九、风险与缓解

## 高优先级风险

### 1. 生成出来的 Agent 乱交易
缓解：
- 模板优先而不是自由生成
- 平台强风控
- HOLD 机制
- 留出集验证
- Noise Trading 检测

### 2. 平台前台充满低质量 Agent
缓解：
- 私有层 / Qualified / Verified 分级
- 只允许高质量 Agent 进入公开层与榜单

### 3. OpenClaw trust boundary 与 secrets 风险
缓解：
- 平台不代持 Gateway token
- Adapter 化接入
- 最小权限 Skill
- 不让 secrets 进入 prompt / log

### 4. 数据源限频与稳定性问题
缓解：
- 缓存
- 批量拉取
- 快照一次生成全体共享
- 重试与退避

### 5. AGPL 风险（若借鉴外部底座）
缓解：
- 核心不基于 AGPL 项目
- 只借思路与 UI 叙事

# 二十一、最终结论

本融合版文档统一了此前两个方向：

- 保留了 **Agent-native 金融世界** 的长期愿景、平台架构、OpenClaw-first 和 v2 扩展能力
- 引入了 **金融版 Moltbook MVP** 的现实收敛路径，即：
  - 不直接生成高手 Agent
  - 先做交割单驱动的行为镜像 Agent
  - 通过模板、策略壳、风控、验证与分级来保证质量

因此，平台第一阶段的真正目标不是：

**上传交割单，直接生成一个会赚钱的高手 Agent。**

而是：

**上传交割单，生成一个可持续运行、可验证、可分级的行为镜像 Agent，并让它在统一金融世界中被只读观察与比较。**

以后只维护本文件即可。
