# 节点式流程图说明

## 1. 文档目的

本文件用于把当前系统写成一条**可执行的节点流水线**。

目标不是画一个漂亮图，而是把下面这些东西说清楚：

1. 每个节点做什么
2. 每个节点输入什么
3. 每个节点输出什么
4. 哪些节点可以并行
5. 哪些节点必须串行
6. 哪些节点后面才能接验证和反思

---

## 2. 总流程

```text
Node 01 事件输入
→ Node 02 事件结构化
→ Node 03 并行功能 Agent 输出
→ Node 04 分歧辩论 / 汇总
→ Node 05 Belief Graph 构建
→ Node 06 三剧本生成
→ Node 07 风险 / 失效 / 拥挤检查
→ Node 08 Dominant Scenario 选择
→ Node 09 报告卡输出
→ Node 10 T+1 / T+3 回填验证
→ Node 11 Reflection Engine
→ Node 12 参与者记忆 / 规则更新
```

---

## 3. 节点说明

### Node 01：事件输入（Event Input）

**作用：**
接收原始事件文本。

**输入：**
- 新闻
- 公告
- 快讯
- 用户手动输入

**输出：**
```json
{
  "event_id": "evt_001",
  "title": "...",
  "body": "...",
  "source": "...",
  "event_time": "...",
  "raw_text": "..."
}
```

**是否可并行：** 否

---

### Node 02：事件结构化（Event Structuring）

**作用：**
把原始文本转成系统可用的结构化事件。

**输入：**
- `raw_text`
- `source`
- `event_time`

**输出：**
```json
{
  "event_type": "supply_shock",
  "entities": [],
  "commodities": [],
  "sectors": [],
  "affected_symbols": [],
  "causal_chain": [],
  "monitor_signals": [],
  "invalidation_conditions": []
}
```

**是否可并行：** 否

**备注：**
第一阶段只支持 1 类事件类型也可以。

---

### Node 03：并行功能 Agent 输出（Parallel Functional Agents）

**作用：**
多个功能 Agent 从不同角度解释事件。

**推荐子节点：**
- Event Structuring Agent（二次校正）
- Industry Transmission Agent
- Sentiment Propagation Agent
- Flow Style Agent
- Risk Falsification Agent
- Policy Interpretation Agent
- Symbol Mapping Agent
- Validation Planner Agent

**输入：**
- `event_structure`
- `market_context`
- `mapping_data`

**输出：**
统一的 `agent_output[]`

**是否可并行：** 是

---

### Node 04：分歧辩论 / 汇总（Debate / Aggregation）

**作用：**
把并行 Agent 的输出变成“看多 / 看空 / 审慎”的分歧结构。

**输入：**
- `agent_outputs`

**输出：**
```json
{
  "bullish_case": [],
  "bearish_case": [],
  "cautious_case": [],
  "unresolved_questions": []
}
```

**是否可并行：** 部分可并行，最终聚合需串行

---

### Node 05：Belief Graph 构建（Belief Graph Builder）

**作用：**
把分歧和参与者立场转成关系层。

**输入：**
- `agent_outputs`
- `debate_state`
- `participant_profiles`（第二阶段）

**输出：**
```json
{
  "consensus_strength": 0.0,
  "dissent_width": 0.0,
  "confidence_dispersion": 0.0,
  "crowding_score": 0.0,
  "falsification_fragility": 0.0,
  "support_clusters": [],
  "opposition_clusters": []
}
```

**是否可并行：** 否

---

### Node 06：三剧本生成（Scenario Engine）

**作用：**
生成：
- 基准剧本
- 乐观剧本
- 风险剧本

**输入：**
- `event_structure`
- `belief_graph`
- `debate_state`

**输出：**
```json
{
  "baseline_scenario": {},
  "bullish_scenario": {},
  "risk_scenario": {}
}
```

**是否可并行：** 否

---

### Node 07：风险 / 失效 / 拥挤检查（Risk Layer）

**作用：**
对三剧本进行压力测试。

**输入：**
- `scenarios`
- `belief_graph`

**输出：**
```json
{
  "dominant_risks": [],
  "invalidation_priority": [],
  "crowding_alerts": []
}
```

**是否可并行：** 否

---

### Node 08：Dominant Scenario 选择

**作用：**
结合三剧本与风险检查，选出当前最值得跟踪的主剧本。

**输入：**
- `scenarios`
- `risk_output`
- `belief_graph`

**输出：**
```json
{
  "dominant_scenario": "baseline",
  "ranking": ["baseline", "risk", "bullish"],
  "why": []
}
```

---

### Node 09：报告卡输出（Report Card）

**作用：**
生成可展示给用户看的单页结果。

**输出内容：**
- 事件摘要
- 核心逻辑链
- 8 类 Agent 观点摘要
- Belief Graph 摘要
- 三剧本
- 关键观察点
- 失效条件
- Dominant Scenario

---

### Node 10：T+1 / T+3 回填验证（Outcome Tracker）

**作用：**
在未来窗口回填真实结果。

**输入：**
- `event_id`
- `dominant_scenario`
- `monitor_signals`

**输出：**
```json
{
  "t1_result": {},
  "t3_result": {},
  "hit_level": "full_hit",
  "latency_score": 0.0,
  "miss_reason": []
}
```

---

### Node 11：Reflection Engine

**作用：**
把验证结果回灌给系统。

**输入：**
- `outcome_tracker_result`
- `agent_outputs`
- `belief_graph`
- `scenarios`

**输出：**
- 哪类 Agent 更准
- 哪类事件结构更容易兑现
- 哪类拥挤更危险
- 哪类失效条件最常成立

---

### Node 12：参与者记忆 / 规则更新

**作用：**
更新长期系统知识。

**更新内容：**
- Participant scores
- Event type prior
- Failure pattern library
- Scenario template weights

---

## 4. 可并行与必须串行的部分

### 可并行
- Node 03 并行功能 Agent
- 某些外部工具数据读取
- 多个验证抓取任务

### 必须串行
- 事件结构化
- Belief Graph 聚合
- 三剧本生成
- Dominant Scenario 选择
- Reflection 结论写回

---

## 5. 第一阶段建议只做的节点

第一阶段只做：

```text
01 事件输入
02 事件结构化
03 并行功能 Agent 输出
04 分歧辩论 / 汇总
05 Belief Graph 构建
06 三剧本生成
07 风险 / 失效检查
08 Dominant Scenario 选择
09 报告卡输出
10 T+1 / T+3 回填验证
```

**第一阶段先不做：**
- 复杂多轮模拟
- 参与者长期记忆
- 自动规则更新
- 完整 ReportAgent 对话

---

## 6. 第二阶段再接入的节点

第二阶段新增：
- Participant Profiles
- Participant Router
- 小规模 Simulation Runner
- Action Logger
- Reflection Engine

---

## 7. 最终作用

这个节点式流程的最大价值，不是“看起来高级”，而是它能让你：

1. 拆任务
2. 并行开发
3. 明确输入输出
4. 后面容易加 TradingAgents 式反思机制
5. 后面容易加 MiroFish 式报告层

一句话总结：

**你不是在写一个大模型问答系统，你是在搭一条事件推演流水线。**
