# 需求分析_1.6：最终版事件参与者优先路线

## 1. 文档定位

本文件是 Versefina 当前阶段的**唯一主需求文档**。

它用于统一并替代此前多条并行主线在执行口径上的分裂，明确后续开发应以“事件驱动的金融参与者预演”作为第一主线推进。

本文件不删除以下历史文档，它们仍然保留参考价值：

- `需求分析_1.1.md`：交割单驱动的行为镜像 Agent
- `需求分析_1.2.md`：金融反身性引擎 / 事件推演系统
- `需求分析_1.3_Aaru增强改造版.md`：市场参与者仿真层增强思路
- `需求分析_1.4.md`：事件驱动主线切换说明
- `需求分析_1.5.md`：TradingAgents 可借鉴工程骨架

但从本版本开始，**对内执行、对外说明、后续 backlog 设计，统一以 1.6 为准**。

---

## 2. 一句话定义

**输入一条金融事件，系统预演不同金融参与者如何卷入、如何扩散、如何形成分歧、什么情况下失效，并在 T+1 / T+3 做后验验证。**

---

## 3. 本版本核心决策

### 3.1 主线正式切换

Versefina 当前阶段的第一主线正式调整为：

```text
事件输入
→ 金融参与者准备层
→ Belief Graph
→ 三剧本
→ 轻量模拟
→ 日志 / 报告
→ T+1 / T+3 验证
```

### 3.2 交割单路线保留但降级

`交割单 -> 个人镜像 Agent` 不取消，但不再作为第一阶段主入口。

它的新定位是：

- 用于补充 `MarketStyleEmbedding`
- 用于抽象 `Archetype Seed`
- 用于校准参与者风格与事件激活关系
- 用于长期分布级验证

换句话说，交割单线是**第二条路线**，不再卡住第一阶段的事件沙盘 MVP。

### 3.3 借鉴 MiroFish，但不照搬

MiroFish 对当前版本真正有价值的，不是完整世界图谱或大规模社会模拟，而是它的流程骨架：

```text
prepare
→ run
→ log
→ report
```

Versefina 只借鉴这条流水线，并将其压缩为**金融事件沙盘版本**：

```text
事件准备
→ 参与者运行
→ 动作日志
→ 复盘报告
```

### 3.4 Aaru 的启发只保留“参与者层”和“分布校准”

本版本延续 `1.3` 的判断：

- 不做通用多智能体调研平台
- 不做大而全金融数字孪生
- 只吸收两层最有价值的增强：
  - 市场参与者仿真层
  - 分布校准层

---

## 4. 为什么现在要这样调整

此前 Versefina 同时存在两条较强主线：

1. 交割单驱动的个人交易镜像 Agent
2. 事件驱动的金融叙事 / 题材扩散预演

这两条主线都成立，但它们的**第一阶段产品效率**不同。

当前优先切换到事件参与者主线，原因如下：

- 更贴近真实高频需求：盘后、盘前、题材发酵、链路扩散、失效判断
- 更容易形成可展示、可验证、可复盘的闭环
- 更适合快速做出类似“事件沙盘”的可见产品
- 更容易和 Belief Graph、三剧本、后验验证形成强闭环
- 交割单路线仍可作为第二阶段的风格母体和校准资产接入

因此，当前版本的重点不再是“AI 帮你复制一个人过去怎么交易”，而是“AI 帮你预演一个事件会由哪些金融参与者推动、如何演化、何时失效”。

---

## 5. 产品本质与边界

### 5.1 当前产品本质

当前产品是：

**单事件驱动的金融参与者预演与题材扩散沙盘系统**

### 5.2 当前产品不是

当前产品不是：

- 自动下单系统
- 实盘券商接入系统
- 高胜率荐股神器
- 完整金融世界仿真平台
- 通用投资顾问
- 自研大模型训练平台

### 5.3 第一版输出不是“买哪个票”

第一版输出应聚焦于：

- 哪条逻辑可能发酵
- 谁构成了推动力量
- 先动什么、后动什么
- 哪些参与者支持、哪些参与者质疑
- 哪些确认信号值得明天重点观察
- 什么情况下这条逻辑会失效

## 6. 首发事件范围

### 6.1 首发白名单

第一版只做以下一类事件：

**涨价链 / 供给冲击 / 产业链传导**

例如：

- 上游原料涨价
- 供给受限
- 限产 / 事故 / 停产
- 出口限制 / 进口约束
- 产业链成本传导

### 6.2 暂不优先

以下事件类型不进入第一版主样板：

- 政策催化
- 利空扩散
- 跨市场复杂宏观事件

### 6.3 为什么首发先选这类

原因：

- 事件结构更清晰
- 因果链更容易结构化
- 更适合维护“商品 -> 产业链 -> 行业 -> 标的池”的映射资产
- 更容易观察谁先动、谁跟随、谁撤退
- 更适合做题材扩散与板块传导分析

## 7. 核心用户与使用场景

### 7.1 核心用户

- A 股短线 / 事件驱动交易者
- 高频看资讯的研究员
- 财经内容创作者
- 中小投研团队
- 题材跟踪型自媒体

### 7.2 核心场景

- 盘后看到涨价或供给冲击消息，判断第二天会如何扩散
- 看到原料端变化，推演上下游谁先受益、谁后受益
- 想提前准备盘前剧本，而不是只看碎片资讯
- 想理解当前主流逻辑由谁构成，为什么容易成立或失效

```text

```

## 9. 系统核心思想

### 9.1 从“新闻总结”升级为“参与者预演”

旧范式：

```text
新闻
→ 总结
→ 利好 / 利空判断
→ 板块 / 个股
```

新范式：

```text
新闻
→ 结构化事件
→ 金融参与者准备
→ 共识 / 分歧聚合
→ 扩散路径预演
→ 剧本生成
→ 轻量模拟
→ 后验验证
```

### 9.2 从“单模型拍结论”升级为“多参与者分歧”

第一版不依赖一个万能模型直接下结论，而是：

- 先把事件结构化
- 再用不同参与者视角产生观点
- 再聚合成共识、分歧、拥挤度、脆弱性
- 再生成多个可验证剧本

### 9.3 从“单次正确”升级为“持续校准”

产品目标不是一次性“猜对”，而是持续积累：

- 哪类参与者在何类事件中更可靠
- 哪类分歧结构更容易演化成主线
- 哪类拥挤结构更容易崩
- 哪类观察信号最适合作为失效条件

## 10. P0 核心对象统一定义

本版本统一以下核心对象作为产品和工程公共语言。

原始事件记录。

最小字段：

- `event_id`
- `title`
- `body`
- `source`
- `event_time`
- `event_type_hint`

关键字段：

- `event_type`
- `entities`
- `commodities`
- `chain_links`
- `sectors`
- `affected_symbols`
- `causal_chain`
- `monitor_signals`
- `invalidation_conditions`

### 10.3 ParticipantBelief

单个参与者视角对事件的结构化判断。

关键字段：

- `participant_family`

- `trigger_conditions`

- `secondary_movers`

### 10.4 ParticipantRoster

- `participants`
- `authority_weights`
- `risk_budgets`

### 10.10 MarketStyleEmbedding

### 10.11 MarketParticipantProfile

从风格向量抽象出来的参与者原型对象。

## 11. 主 API 族

第一阶段 API 统一围绕事件沙盘组织：

- `POST /api/v1/events`
- `POST /api/v1/events/{event_id}/prepare`
- `POST /api/v1/events/{event_id}/run`
- `GET /api/v1/events/{event_id}/card`
- `GET /api/v1/events/{event_id}/report`
- `POST /api/v1/events/{event_id}/outcomes`
- `GET /api/v1/events/{event_id}/why`

接口语义：

- `events`：写入原始事件
- `prepare`：完成结构化、参与者激活、Belief 准备
- `run`：执行轻量模拟并产生日志
- `card`：返回事件推演卡
- `report`：返回复盘报告
- `outcomes`：回填 T+1 / T+3 结果
- `why`：基于过程数据做追问解释

## 12. MVP 功能模块

### 12.1 Event Ingestion

负责原始事件接收、去重、入库、状态初始化。

### 12.2 Event Structuring

负责把原始文本抽成结构化事件 JSON。

### 12.3 Participant Preparation

负责把结构化事件转成可运行的参与者 roster，包括：

- 8 类参与者输出协议
- 主变体与参与者家族
- authority weight
- risk budget
- 证据与失效条件标准化

### 12.4 Belief Graph

负责把参与者 beliefs 聚合成可计算中间状态，而不是保留一堆文本。

### 12.5 Scenario Engine

负责生成 `base / bull / bear` 三剧本，以及关键观察点和失效条件。

### 12.6 Lightweight Simulation

负责进行 3 到 5 轮轻量推演，回答：

- 谁先动
- 谁跟随
- 谁撤退
- 哪个信号导致状态切换

### 12.7 Logging & Reporting

负责写入动作日志、状态快照、事件推演卡和复盘报告。

### 12.8 Outcome Tracker

负责 T+1 / T+3 回填、dominant scenario 判分、参与者可靠性统计。

### 12.9 Statement-to-Style Assets

第二阶段接入交割单后，生成：

- 行为特征
- 风格向量
- 风格母体
- 参与者激活校准规则

## 14. 验收与测试标准

### 14.1 P0 必须满足

- 样板事件使用“涨价链 / 供给冲击 / 产业链传导”
- 至少能稳定输出：
  - 结构化事件
  - 8 类参与者立场
  - Belief Graph
  - 三剧本
  - 一次轻量模拟
  - 一次复盘报告

### 14.2 稳定性要求

- 同一事件重复运行时：
  - `EventStructure` 不得漂移
  - 参与者 schema 不得漂移
  - Belief Graph 指标不得随意变形
- 允许置信度微调，但不允许字段不稳定

### 14.3 降级要求

证据不足时必须允许：

- `neutral`
- `watch`
- `insufficient_evidence`

不允许强行给出过度结论。

### 14.4 轻量模拟要求

必须能解释：

- 首动对象
- 跟随对象
- 撤退对象
- 关键失效点

### 14.5 后验验证要求

T+1 / T+3 回填后，必须给出：

- `dominant_scenario`
- 参与者可靠性变化
- 失败原因

不能只回写价格结果。

### 14.6 第二阶段要求

交割单接入后，必须能把行为特征映射到参与者风格，而不是只生成一个孤立镜像 Agent。

## 15. 对旧路线的迁移原则

### 15.1 旧任务资产不覆盖

现有 `tasks/` 中旧的 Sprint 1 到 Sprint 9 目录不直接覆盖。

原因：

- 它们反映了此前阶段的规划历史
- 已经存在人工编辑痕迹
- 后续需要保留为参考与对照

### 15.2 新路线单独建档

从 1.6 开始，新的主路线使用新的 sprint/story 资产：

- 新的 sprint 总览文件
- 新的 sprint 目录
- 新的 35 个 story YAML

### 15.3 world 底座复用原则

现有 `market_world` 不要求先演进为完整撮合世界后，事件线才能开始。

当前只复用其：

- 交易日历
- 时间轴
- 世界快照概念

完整世界、撮合、账本仍属于后续扩展底座，不是 1.6 P0 的硬前置。

## 17. 最终结论

Versefina 1.6 正式确定：

### 主线

**金融事件驱动的参与者预演与题材扩散沙盘**

### 方法
