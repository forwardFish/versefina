# 需求分析_1.5.md

## 1. 文档定位

本版本基于 **需求分析_1.4** 继续增强，目标不是改主线，而是新增一层：

**把 TradingAgents 里值得借鉴、可复制、可改造的工程骨架，翻译成当前“金融叙事 / 题材扩散预演”系统的实现参考。**

本版本回答 4 个问题：

1. TradingAgents 到底是什么，适不适合当前主线
2. TradingAgents 里哪些模块可以直接借鉴
3. 哪些东西不能照搬，否则会把产品带偏
4. 如何把 TradingAgents 的工程结构，映射到当前项目中

---

## 2. 与 1.4 的关系

### 2.1 不变的内容

本版本不改变 1.4 的核心主线：

- 主线仍然是 **金融叙事 / 题材扩散预演**
- 第一阶段仍然以 **事件驱动** 为主入口
- 第一版仍然不做自动交易、不做实盘、不做完整金融世界、不做训练大模型
- 交割单仍然属于 **后续参与者画像 / 风格母体 / 分布校准** 增强层

### 2.2 本版本新增的内容

本版本新增：

- TradingAgents 的可借鉴模块清单
- 当前系统与 TradingAgents 的模块映射关系
- 可复用的工程设计模式
- 可复制的代码组织方式
- 不可照搬的部分与风险提醒
- 基于 TradingAgents 思路优化后的开发阶段建议

## 3. 一句话定义（1.5 版）

**在 1.4 主线基础上，借鉴 TradingAgents 的“多角色分工 + 辩论 + 配置化 + 反思记忆 + 图编排”工程方式，构建一个事件驱动的金融叙事 / 题材扩散预演系统。**

## 4. 对 TradingAgents 的总判断

### 4.1 它是什么

TradingAgents 更像一个：

**多分析师 + 多空辩论 + 风控审批 + 最终交易决策** 的交易研究框架。

它的核心输入通常是：

- 某只股票
- 某个交易日期

它的核心输出通常是：

- 最终交易决策
- 风险评估
- 决策过程日志

### 4.2 它对当前项目的价值

TradingAgents 对当前项目的价值，不在于“直接变成我们的产品”，而在于它提供了 5 个很重要的工程启发：

1. **多角色分工结构**
2. **辩论式分歧收敛机制**
3. **图式流程编排（Graph Orchestration）**
4. **可配置化系统设计**
5. **反思与记忆更新机制**

### 4.3 它不能直接当主框架的原因

它的原始设计更偏向：

- 单股分析
- 投研团队模拟
- 交易决策输出

而当前项目的主线是：

- 事件驱动
- 题材扩散
- 市场参与者分歧
- 三剧本推演
- T+1 / T+3 后验验证

因此：

**TradingAgents 可以当“骨架参考”，不能当“主产品模板”。**

---

## 5. 当前项目与 TradingAgents 的本质差异

| 维度 | 当前项目 | TradingAgents |
|---|---|---|
| 核心输入 | 一条金融事件 | 一只股票 + 日期 |
| 核心问题 | 题材会不会发酵、由谁推动、如何失效 | 这只股票该不该交易 |
| 核心输出 | 三剧本、Belief Graph、观察点、失效条件 | 最终交易决策 |
| 研究对象 | 事件 → 板块 → 标的 → 参与者 | 单股投研 |
| 核心能力 | 叙事扩散推演 | 多分析师交易框架 |
| 验证方式 | T+1 / T+3 剧本验证 | 收益 / 风险 / 回测表现 |

结论：

**它不是你的主系统，但它的很多“工程件”非常值得借。**

## 6. TradingAgents 里可以直接借鉴的东西

下面这些，是当前项目可以明确借鉴的部分。

---

### 6.1 借鉴点一：角色分层

TradingAgents 的角色层次很清楚：

- 分析师层
- 多空辩论层
- 交易决策层
- 风控层
- 记忆反思层

当前项目可以改造为：

#### 分析层
- 事件结构化 Agent
- 产业链传导 Agent
- 情绪扩散 Agent
- 资金风格 Agent
- 风险证伪 Agent

#### 分歧层
- 看多派
- 看空派
- 审慎派

#### 聚合层
- Belief Graph Builder
- Scenario Engine

#### 风控层
- Invalidation Checker
- Crowding Risk Checker

#### 复盘层
- Outcome Tracker
- Reflection Engine

也就是说：

**借它的“分层”，不要借它的“原角色名字”。**

---

### 6.2 借鉴点二：辩论式分歧收敛

TradingAgents 的一个核心思路是：

不是一个模型直接给答案，而是：

- 多头给理由
- 空头给理由
- 中间再有整合或裁决

这对当前项目非常有用。

当前系统中，8 类 Agent 原本就容易出现分歧，所以完全可以借这个机制：

- 先让不同 Agent 独立输出
- 再进入辩论 / 汇总阶段
- 再由 Scenario Engine 生成三剧本

这会比“直接平均”更像真实市场。

---

### 6.3 借鉴点三：Graph Orchestration（图式流程编排）

TradingAgents 用图来编排流程，而不是把所有逻辑写成一条很长的链。

这个思路非常适合当前系统。

当前项目推荐改造成：

```text
EventInput
→ EventStructuring
→ AgentOutputs
→ Debate / Aggregation
→ BeliefGraph
→ ScenarioEngine
→ OutcomeTracker
→ ReportAgent
```

推荐把当前系统也做成“节点式”流程，而不是一坨大函数。

---

### 6.4 借鉴点四：配置化设计

TradingAgents 里很多东西不是写死在代码里，而是放到 config 里。

当前项目也应该这么做：

#### 推荐配置化的内容
- 事件类型白名单
- 8 类 Agent 权重
- 剧本数量
- 辩论轮数
- 验证窗口（T+1 / T+3 / T+5）
- 市场参与者模板
- 数据源开关
- 风险阈值

换句话说：

**把“研究策略”做成配置，而不是硬编码。**

---

### 6.5 借鉴点五：记忆与反思机制

TradingAgents 里有一个很有价值的思想：

系统不是只做一次判断，而是：

- 跑完
- 看结果
- 再更新记忆

当前系统也应该做这件事。

可以新增：

- 哪类 Agent 在什么事件上更准
- 哪类共识结构更容易兑现
- 哪类拥挤结构更容易失效
- 哪类失效条件最常提前触发

也就是说：

**Outcome Tracker 之后，应该有 Reflection Engine。**

---

### 6.6 借鉴点六：日志全量保存

TradingAgents 会保存过程状态，这一点非常值得抄。

当前系统建议保存：

- 原始事件文本
- 结构化事件 JSON
- 8 Agent 输出 JSON
- Belief Graph 快照
- 三剧本结果
- 每轮模拟动作日志
- T+1 / T+3 验证结果
- 最终复盘报告

原因：

- 方便复盘
- 方便调试
- 方便回测
- 方便以后做校准

---

### 6.7 借鉴点七：ToolNode 思维

TradingAgents 会把不同数据能力封装成工具节点。

当前项目也可以明确拆成工具层：

- `event_source_tools`：快讯 / 新闻 / 公告提取
- `industry_mapping_tools`：产业链映射
- `symbol_mapping_tools`：板块与标的映射
- `validation_tools`：T+1 / T+3 校验
- `market_state_tools`：行情与板块热度提取

这样后面更容易替换数据源。

---

### 6.8 借鉴点八：双模型策略

TradingAgents 有“快模型 + 深模型”思路。

当前项目也可以这样：

#### 快模型
用于：
- 文本清洗
- 事件结构化
- 标签提取
- 简单映射

#### 深模型
用于：
- Agent 分歧生成
- 三剧本生成
- 风险解释
- 报告生成

这样可以降低成本，也更稳定。

## 7. 不要直接照搬的东西

下面这些，必须明确写进文档，避免项目被带偏。

---

### 7.1 不照搬“单股中心”

TradingAgents 的输入方式更偏：

```text
company_name + trade_date
```

当前项目不能照搬。

当前项目必须坚持：

```text
event_text / event_id
```

也就是说：

**当前系统的核心对象是“事件”，不是“股票”。**

---

### 7.2 不照搬“最终交易决策”当主目标

TradingAgents 最后想给：

- 买 / 卖 / 持有
- 风险确认

当前项目第一版不能把目标设成这个。

当前系统第一版的目标应该继续保持为：

- 基准剧本
- 乐观剧本
- 风险剧本
- 关键观察点
- 失效条件
- 后验验证

否则系统会从“题材扩散预演”退化成“单股投顾框架”。

---

### 7.3 不照搬它的分析师类别

TradingAgents 常见分析维度是：

- 基本面
- 新闻
- 技术面
- 情绪

当前项目虽然也会参考这些，但主线应改成：

- 事件结构
- 产业链传导
- 市场参与者预期
- 资金风格反应
- 风险证伪路径

换句话说：

**你更关心“题材如何扩散”，而不是“单股基本面完整投研”。**

---

### 7.4 不照搬它的数据源结构

TradingAgents 天然更适合：

- 美股
- 通用股票基本面
- Alpha Vantage / yfinance 之类数据源

当前项目后续必须优先接适合自身市场的事件数据、板块数据、公告与快讯数据。

所以只能借“数据源抽象方式”，不能照搬“数据源本身”。

---

### 7.5 不照搬它的收益评价体系

TradingAgents 最终更容易用：

- 收益
- Sharpe
- 回撤

当前项目第一版更该评价：

- 剧本命中率
- 提前量
- 失效识别速度
- 参与者判断稳定性
- 分歧结构解释力

也就是说：

**评价指标不同，系统目标就不同。**

---

## 8. 当前项目对 TradingAgents 的映射改造表

| TradingAgents 组件 | 当前项目改造版本 | 是否可借 |
|---|---|---|
| Analyst Team | 事件解构 / 产业链 / 情绪 / 风险 Agent | 可借 |
| Bull / Bear Researchers | 看多 / 看空 / 审慎派辩论层 | 可借 |
| Trader | Scenario Engine / Dominant Scenario Selector | 改造后可借 |
| Risk Manager | Invalidation Checker / Crowding Risk Checker | 可借 |
| Portfolio Manager | Scenario Finalizer / Report Gatekeeper | 改造后可借 |
| ToolNodes | 数据源抽象层 | 可借 |
| Graph Orchestration | 事件流程节点图 | 强烈建议借 |
| Reflection / Memory | Outcome Tracker + Reflection Engine | 强烈建议借 |
| Single-stock propagate | Event propagate | 必须改写 |
| Final trade decision | 三剧本 / 观察点 / 失效条件 | 必须改写 |

## 9. 当前版本新增的系统结构建议

基于 1.4 主线与 TradingAgents 的骨架，推荐当前系统升级为以下结构：

```text
app/
  api/
    events.py
    scenarios.py
    outcomes.py
    reports.py

  core/
    config.py
    llm_router.py
    graph_setup.py

  nodes/
    event_structuring_node.py
    chain_mapping_node.py
    sentiment_node.py
    market_participant_node.py
    risk_node.py
    debate_node.py
    belief_graph_node.py
    scenario_node.py
    outcome_node.py
    reflection_node.py
    report_node.py

  services/
    event_service.py
    participant_service.py
    belief_service.py
    scenario_service.py
    outcome_service.py
    report_service.py

  tools/
    event_source_tools.py
    mapping_tools.py
    validation_tools.py
    market_state_tools.py

  memory/
    agent_memory.py
    event_memory.py
    outcome_memory.py

  storage/
    models.py
    repositories.py
```

这个目录设计的意思是：

- 借 TradingAgents 的图编排和模块化
- 但把输入对象从 `ticker` 改成 `event`
- 把输出对象从 `trade decision` 改成 `scenario report`

## 10. 当前系统新增的可复制实现模式

### 10.1 模式一：节点式执行

每一步都做成一个 Node：

- 输入固定
- 输出固定
- 可单独测试
- 可单独替换

这比把所有逻辑写在一个大函数里更适合后续增长。

---

### 10.2 模式二：状态对象贯穿全流程

推荐定义统一状态对象，例如：

```json
{
  "event": {},
  "structured_event": {},
  "agent_outputs": [],
  "debate_state": {},
  "belief_graph": {},
  "scenarios": [],
  "simulation_logs": [],
  "outcome": {},
  "report": {}
}
```

这其实就是借 TradingAgents 的“stateful graph”思路。

---

### 10.3 模式三：结果回灌

当前系统建议所有后验结果都能反哺：

- 参与者权重
- 拥挤阈值
- 失效触发规则
- 事件类型表现历史

这样系统不是静态规则机，而是会逐步校准。

## 11. 1.5 版新增阶段路线图

### Phase 1：保持 1.4 的 MVP 主线

只做：

- Event Ingestion
- Event Structuring
- 8 Agent Outputs
- Belief Graph
- 三剧本
- Outcome Tracker

目标：

**先跑通事件沙盘。**

---

### Phase 2：引入 TradingAgents 风格的分层与辩论

新增：

- Debate Node
- Risk Node
- Scenario Finalizer
- Reflection Engine

目标：

**从“8 个分析器”升级到“分层协作系统”。**

---

### Phase 3：引入市场参与者家族

新增：

- Participant Profiles
- Family / Style 模板
- 风险预算 / 反应规则
- 小规模模拟循环

目标：

**从静态剧本生成，升级为参与者驱动的动态推演。**

---

### Phase 4：交割单接入与分布校准

新增：

- 行为特征提取
- 风格母体
- 参与者分布修正
- 长期记忆更新

目标：

**让市场参与者不是手工编出来，而是逐步吸收真实行为。**

## 12. 1.5 版新增验收标准

### 12.1 工程层面

- 节点式流程可单独测试
- 关键输入输出固定 JSON 协议
- 关键状态可落盘
- 所有结果可复盘

### 12.2 业务层面

- 同一条事件能输出稳定的分歧结构
- 同一类事件能积累历史记忆
- 不同参与者风格能形成不同剧本路径
- Outcome Tracker 能驱动后续校准

### 12.3 产品层面

- 不被带偏成“单股交易系统”
- 保持“事件驱动 / 题材扩散预演”主线
- TradingAgents 的引入只增强工程能力，不改变产品本质

---

## 13. 当前阶段最值得落地的结论

### 13.1 你应该复制的

- 多角色分层
- 节点式流程图
- 辩论机制
- 配置化
- 记忆与反思
- 全量日志
- 状态驱动执行

### 13.2 你不应该复制的

- 单股为中心
- 最终买卖决策导向
- 美股原生数据逻辑
- 以收益为唯一指标
- 原始分析师分类

---

## 14. 1.5 版最终结论

TradingAgents 对当前项目最重要的价值不是“让你直接照搬一套交易系统”，而是提供了一套：

**多角色、可配置、可反思、可编排、可落盘的 Agent 工程骨架。**

因此，1.5 版的核心策略是：

**保留 1.4 的产品主线，吸收 TradingAgents 的工程方法。**

换句话说：

- **产品灵魂仍然来自 1.4**
- **工程骨架可以借 TradingAgents**
- **交割单增强层继续放在后面**
- **第一版仍以事件沙盘为主**
